# Security — data plane

Everything that touches persisted or streamed data: the append-only audit log,
per-job secret redaction, stdout/stderr scrubbing, the PostgreSQL-authoritative
log store, and the analytics PII posture.

## 1. Audit logging

Every security and operational event is written to an append-only BadgerDB
log. Design invariant: never updated, never deleted in normal operation.

### Event types

| Event | Trigger |
|---|---|
| `node_register` | Node registers with coordinator |
| `node_revoke` | Node certificate revoked via API |
| `job_submit` | Job submitted via REST API |
| `job_state_transition` | Job status changed (any transition) |
| `auth_failure` | JWT missing, expired, revoked, or invalid |
| `rate_limit_hit` | Per-node rate limit exceeded |
| `security_violation` | Seccomp or OOMKilled reported by node |
| `coordinator_start` / `coordinator_stop` | Process lifecycle |
| `analytics.query` | Authenticated call to `/api/analytics/*`; details carry `endpoint`, `from`, `to`, `actor` |
| `env_denylist_reject` / `env_denylist_override` | See [runtime.md § Env denylist](runtime.md#2-dangerous-env-denylist-feature-25) |
| `secret_revealed` / `secret_reveal_reject` | See [§ 3 Secret env redaction](#3-secret-env-redaction-feature-26) |
| `operator_cert_issued` / `operator_cert_missing` / `operator_cert_revoked_used` | See [crypto.md § Operator-cert revocation](crypto.md#4-operator-cert-revocation-feature-31) |
| `token_cert_cn_mismatch` | See [operator-auth.md § Token binding](operator-auth.md#2-token--cert-cn-binding-feature-33) |
| `webauthn_*` | See [operator-auth.md § WebAuthn](operator-auth.md#3-webauthn--fido2-feature-34) |
| `authz_deny` | Every 403 from the policy engine (feature 37) |
| `group_*`, `resource_shared`, `resource_share_revoked` | Feature 38 share mutations |
| `service.ready` / `service.unhealthy` | Edge-triggered service readiness |
| `secretstore_rotate` | Feature 30 KEK rotation sweep |

### Storage

- Key: `audit:{timestamp_nanos}:{event_id}` — time-ordered
- Default TTL: 90 days; set `HELION_AUDIT_TTL=0` to disable expiry
- Never updated, never deleted in normal operation

### Query API

```bash
# Paginated events
curl -H "Authorization: Bearer $ROOT_TOKEN" \
  "https://coordinator:8443/audit?page=1&size=50"

# Filter by type
curl -H "Authorization: Bearer $ROOT_TOKEN" \
  "https://coordinator:8443/audit?type=job_submit"

# Count by type
curl -H "Authorization: Bearer $ROOT_TOKEN" \
  "https://coordinator:8443/audit" | jq '.events[] | .type' | sort | uniq -c
```

Response:

```json
{
  "events": [
    {
      "id": "event-123",
      "timestamp": "2026-04-10T12:34:56Z",
      "type": "job_submit",
      "actor": "root",
      "details": { "job_id": "job-xyz", "command": "echo" }
    }
  ],
  "total": 100, "page": 1, "size": 50
}
```

Audit events always carry both forms after feature 35: `Event.Actor` stays the
legacy bare string (user ID, "system", "unknown") for back-compat;
`Event.Principal` + `Event.PrincipalKind` carry the typed identity. Dashboards
filter on the typed fields; pre-feature-35 tooling keeps reading `Actor`.

## 2. Actor attribution

When a request carries a valid JWT, `claims.Subject` is extracted and recorded
as the `actor` in any audit events generated by that request. Unauthenticated
paths record `actor = "anonymous"`.

From feature 36 onward, create-path audit events also include a
`resource_owner` detail alongside `actor`. Reviewers can distinguish "who did
it" (actor) from "who owns the target" (resource_owner) — identical for
user-driven creates, divergent when a service principal acts on a user-owned
resource.

## 3. Secret env redaction (feature 26)

### Response-path redactor

`SubmitRequest` carries a `secret_keys: string[]` list that names env keys
whose values must be redacted on every response path. The coordinator keeps
plaintext in `Env` (the runtime needs it to dispatch) but wraps every response
build through `redactSecretEnv(env, secretKeys)`. Applied on:

- `GET /jobs/{id}` (single job)
- `GET /jobs` (paginated list)
- `POST /jobs` + `POST /workflows` on the 201 response body
- `?dry_run=true` responses on all submit paths
- `GET /workflows/{id}` child-job env

The replacement string is the literal `[REDACTED]`. The redactor returns a
fresh copy of the env map; mutating a response map never pollutes the stored
record.

### Validation at submit

- Every key in `secret_keys` MUST appear in `env`. A flag on a non-existent
  key is rejected with 400 — either a typo or a probe for which names the
  server silently accepts.
- No duplicates; no empty strings; list capped at 32 entries (the overall
  env map caps at 128).

### Audit invariants

- `job_submit` events carry a `secret_keys` detail listing the KEY NAMES.
  Values are NEVER in audit details.
- `env_denylist_reject` / `env_denylist_override` (see [runtime.md](runtime.md))
  follow the same value-free policy.

### Reveal-secret endpoint

`POST /admin/jobs/{id}/reveal-secret` is the single audited path by which an
operator can read back a declared secret value.

Body:

```json
{ "key": "HF_TOKEN", "reason": "on-call debug of HF model load" }
```

- **Admin role only.** `adminMiddleware` runs first; `node`-role tokens get
  403 before the handler sees the request.
- **Rate-limited 1 reveal / 5 s (burst 3).** Tighter than `/admin/tokens`
  because every call exposes plaintext. A compromised admin token cannot
  bulk-dump the coordinator's secret inventory before audit triggers detection.
- **Reason field mandatory**, non-empty, trimmed, ≤512 bytes, NUL-free. Lands
  verbatim in the audit detail.
- **Key must be declared secret on THAT job.** Reading a non-secret env value
  via this endpoint returns 404 — not a generic env reader.
- **Audit written BEFORE response.** `secret_revealed` persists before the
  plaintext enters the response body. A downed audit sink yields 500 and no
  leak — fail-closed on accountability.
- **Every reject is audited too.** Unknown-job, not-declared-secret,
  malformed-body, missing-reason paths each emit `secret_reveal_reject` with
  actor + reason-for-reject. Probe sweeps show loud in the audit stream.
- **Response body carries an audit notice.** `audit_notice` reminds the
  operator on-screen that the reveal was logged.
- **Storage no longer holds plaintext** (feature 30 — see
  [crypto.md § Encrypted env storage](crypto.md#5-encrypted-env-storage-feature-30)).
- **stdout leaks mitigated by feature 29** — see [§ 4](#4-stdout-scrubbing-feature-29).

Spec reconciliation: [planned-features/implemented/26-secret-env-vars.md](../planned-features/implemented/26-secret-env-vars.md).

## 4. stdout scrubbing (feature 29)

Feature 26 closes "operator reads env via `GET /jobs/{id}`". It does not close
"operator reads the job's stdout/stderr and finds the plaintext there because
the job printed it":

```python
print(f"Auth: Bearer {os.environ['HF_TOKEN']}")  # common debug
```

Feature 29 wraps the log store in `logstore.ScrubbingStore` — a decorator
that replaces every occurrence of every declared secret VALUE with the
literal `[REDACTED]` before the chunk lands in BadgerDB. A second pass runs
at response-build time on `GET /jobs/{id}/logs` as belt-and-braces for
chunks that landed before the decorator was wired (rolling deploy edge
case).

### Safety properties

- **Same substitution at both sinks.** The Badger append path is decorated;
  the feature-28 analytics mirror (`events.Bus` → PG) is scrubbed in the
  gRPC `StreamLogs` handler before publish. PG cannot become a
  side-channel around the decorator — both sinks see the same redacted
  bytes.
- **Empty-value DoS guard.** A secret declared with an empty value would
  otherwise match every byte position; the scrubber skips zero-length
  secrets.
- **Idempotent.** Running the scrubber twice produces the same output.
- **Fail-open on store lookup.** A secrets-lookup error (job terminal +
  evicted) returns `(nil, false)` — no scrubbing, but the chunk still
  persists. Prefer "not dropping logs" over a perfect redact; the
  response-path pass gives a second chance.
- **Per-job RBAC on `GET /jobs/{id}/logs`.** Feature 37's authz check gates
  log reads the same as job reads. Pre-feature-29 this endpoint had no
  per-job RBAC.

### Known limitations

- **Chunk-boundary miss.** If a secret value lands split across two write
  chunks, neither chunk matches on its own. Response-path redactor catches
  it at GET time if it concatenates first — current implementation scrubs
  per-chunk, so the boundary case remains. Operators worried about this
  should use structured JSON env delivery.
- **Malicious transform.** A script that base64-encodes or xors its own
  token before printing is not caught; the coordinator does not claim to
  sandbox against "operator chose to run a malicious job".
- **Regex-based detection** of undeclared secrets (e.g. AWS key IDs by
  shape) is explicitly out of scope — false-positives erode operator trust.
  Declare the secret at submit time; the scrubber takes it from there.

Spec: [planned-features/implemented/29-stdout-secret-scrubbing.md](../planned-features/implemented/29-stdout-secret-scrubbing.md).

## 5. Job log persistence (feature 28 — PG-authoritative)

Per-job stdout/stderr lives in two stores with well-defined roles:

- **PostgreSQL `job_log_entries`** — authoritative long-term home. Never
  pruned by the analytics retention cron. A log line from 6 months ago is
  still queryable via `GET /api/analytics/job-logs?job_id=…`.
- **BadgerDB `log:` prefix** — short-term live-tail cache. 7 d TTL by
  default; individual entries freed sooner by the reconciler once PG
  confirms them.

### Reconciler safety contract

- `SELECT 1 FROM job_log_entries WHERE (job_id, seq) = ($1, $2)` per
  candidate; deletion happens only on a positive hit.
- `MinAge` gate (default 5 min, `HELION_LOGSTORE_RECONCILE_MIN_AGE_MIN`)
  protects just-landed chunks from racing the sink's batched flush.
- PG query errors never cascade into deletions. Next reconciler tick
  retries cleanly.
- Opt-in via `HELION_LOGSTORE_RECONCILE`. Operators who prefer dual-copy
  forever leave it off; the Badger TTL still frees space eventually, the
  PG copy remains permanent.
- `TestReconciler_ConfirmedAreDeletedUnconfirmedKept`,
  `TestReconciler_PGQueryError_NoDeletes`, and
  `TestReconciler_AgeGate_SkipsYoung` guard the contract against
  regressions.

### Trade-offs

Buys: Badger stays small on high-log clusters; log history persists as
long as PG does (no forced 7 d ceiling); live-tail UX via Badger's prefix
scan stays fast for in-flight jobs.

Does not buy: a replacement for the audit log (logs are whatever the job
printed, not accountability records). A PG outage during a live job means
Badger keeps everything until PG comes back; if Badger's TTL fires first,
chunks are lost — same as pre-28.

## 6. Analytics PII mode (feature 28)

| Threat | Mitigation |
|---|---|
| Analytics database compromised; attacker reads JWT subjects out of a PG dump | `HELION_ANALYTICS_PII_MODE=hash_actor` writes `sha256(salt \|\| actor)` into every analytics `actor` column. Dashboards still group by hash; audit log remains authoritative for accountability. |
| Analytics database fills indefinitely | `HELION_ANALYTICS_RETENTION_DAYS` default 0 (disabled) because PG is the intended long-term store. Operators set a positive value to enable the cron; every feature-28 table is pruned EXCEPT `job_log_entries` (see § 5). Audit log in BadgerDB is untouched regardless. |
| Secret env values sneak into analytics tables | `submission_history` stores only `resource_id` (a ULID), never the submit body. Command/args/env live in the audit log under the matching `job_id`. Feature 26's redaction applies to the audit store too — an analytics dump is never a secret-exposure event. |

## 7. ML artifact attestation (features 12, 13)

| Threat | Mitigation |
|---|---|
| Compromised node reports a cross-job artifact URI (claims job A's bytes live under job B's prefix) | `attestOutputs` rejects any `ResolvedOutputs` URI that doesn't match `<scheme>://<bucket>/jobs/<reporting-job-id>/<local_path>`. A compromised node can only report URIs under its own running job's prefix. |
| Compromised node reports an undeclared output name (invents names the downstream resolver would accept) | `attestOutputs` cross-checks every reported `Name` against `Job.Outputs` (the submit-time declaration); undeclared names are dropped. Guarded by AUDIT 2026-04-15-05. |
| Compromised node serves tampered artifact bytes under a valid URI | Downstream node's Stager runs `artifacts.GetAndVerify` — reads into a capped buffer, hashes, returns `ErrChecksumMismatch` if the digest doesn't match the SHA-256 committed with the URI. Catches leaked S3-creds tamper, bit rot, and store-side MITM. |
