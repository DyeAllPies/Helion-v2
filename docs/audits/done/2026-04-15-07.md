## Test Gaps

### ~~T1 â€” `ml.resolve_failed` event emit had no observer test~~ âś… Fixed
**File:** `internal/cluster/dispatch.go:218` â€” the
`d.jobs.publishEvent(events.MLResolveFailed(...))` call inside the
resolver-failure branch.

Feature 18's step-3 follow-up added an
`events.TopicMLResolveFailed` topic and wired the dispatch loop
to emit one when the resolver returns an error. The existing
`TestDispatchLoop_Step3_ResolveFailure_TransitionsToFailed` test
proves the job transitions to Failed but never subscribes to
the bus to confirm the event actually fires. That's the worst
kind of silent regression: the test passes, the user-visible
outcome (job Failed) is correct, but the dashboard stops
surfacing broken pipelines because the event never hits the bus.

A future refactor that merged the transition-and-emit into a
single helper and missed the emit half would slip past every
existing test.

**Fix:**
[`dispatch_artifact_resolve_test.go:TestDispatchLoop_Step3_ResolveFailure_EmitsMLResolveFailedEvent`](../../../internal/cluster/dispatch_artifact_resolve_test.go)
subscribes to `events.TopicMLResolveFailed` **before** starting
the dispatch loop (so the publish lands in the buffered channel
rather than being dropped), constructs the same bad-workflow
shape the transition-test uses, drives the upstream to Completed
with no ResolvedOutputs, waits â‰¤ 3 s for the event, and asserts
the payload carries the four fields the dashboard's Pipelines
row needs: `job_id`, `workflow_id`, `upstream`, `output_name`,
plus a non-empty `reason` string.

## Items considered and dismissed (not findings)

| Item | Why not |
|------|---------|
| Resolver idempotency: calling `ResolveJobInputs` twice on an already-rewritten job | The existing `TestResolveJobInputs_NoFromRefs_ReturnsSamePointer` case covers this â€” after the first resolver pass the inputs have no `From` field, and a second pass sees no `From` refs and returns the same pointer. Pointer-equality assertion is the alarm; the rewrite is idempotent by structure. |
| End-to-end workflow artifact chain through real Stager against live MinIO | Straddles features 12 / 13 / 19. Feature 12's audit pass added `TestLiveS3Stager_UploadsOutputOnFinalize` which covers the upload half; the resolver half is structural (reads `Job.ResolvedOutputs`, rewrites Inputs, persists). Adding a second live-MinIO test that additionally exercises the resolver would duplicate feature 12's proof without exercising new code â€” the resolver's own unit tests are sufficient because the join is pure. The full end-to-end chain **is** covered by feature 19's iris demo when it runs; no need for a parallel integration test. |
| Concurrent dispatch of multiple downstream jobs referencing the same upstream | The dispatch loop is single-threaded per iteration (`dispatchPending` walks all pending jobs in a single goroutine). Two downstream jobs with `from: a.OUT` would both be iterated in the same tick; the resolver's read of `Job.ResolvedOutputs` is a lock-free snapshot after persistence. No concurrent mutation possible by design. |
| Workflow retry of a downstream that failed on resolve | When a resolver-failed job retries, it goes back through `dispatchPending`, which re-runs `ResolveJobInputs`. If the upstream's state is unchanged (still missing the output), the second pass fails the same way. If the upstream somehow fixed itself, the second pass succeeds. Either way the behaviour is the one the existing resolver-failure + job-transition tests already cover; the retry loop adds no new branch. |
| `UpdateResolvedInputs` rollback on persistence failure | Already covered by `TestUpdateResolvedInputs_PersistFailure_RollsBack` in `job_update_inputs_test.go`. |
| SHA-256 propagation from upstream â†’ downstream resolved input | Already covered by the `SHA256` field assertion at `workflow_resolve_test.go:94`. |
| Last-dot split rule for `from: a.b.c.OUT` references | Already covered by `TestSplitFromRef` (line `{"a.b.c.D", "a.b.c", "D"}` case). |
| DAG validator rejects non-ancestor references | Already covered by `dag_from_refs_test.go:TestValidateFromReferences_NonAncestor`. |
| DAG validator rejects unknown upstream / unknown output / malformed shape | All three covered by dedicated tests in `dag_from_refs_test.go`. |
| `validateArtifactBindingsCtx` rejects `From` on plain-job submits | Covered by `TestValidateSubmitRequest_PlainJobRejectsFrom`. |
| Mixed URI + From on a single input | Covered by `TestValidateArtifactBindingsCtx_URIAndFromMutuallyExclusive`. |
