Pass 13 predicted saturation at one pass. This pass applied the

every package's tests, then drill the invariants that underlie

the entire `classifyUnschedulable` classification depends on —
unpinned by any test.
**Overall Risk Level:** LOW.

Feature 18's `job.unschedulable.reason` classification has
three values distinguishing three operational fixes:

- `no_healthy_node` — wait, retry (transient cluster state)
- `no_matching_label` — add a matching node or change selector
- `all_matching_unhealthy` — restart the stale matching nodes

The third reason is what distinguishes "wrong labels" from
"right labels, dead nodes." It depends on a single Registry
invariant: **`Snapshot()` must include stale/unhealthy nodes
while `HealthyNodes()` excludes them**. Without the divergence,
`classifyUnschedulable` can't tell the difference and always
returns `no_matching_label`.

Pass 13 focused on the HTTP-layer of feature 18's unique code.
This pass went deeper into the Registry invariants that feature
18's event payloads are built from. Found one:
`Snapshot().includes(staleNodes) && HealthyNodes().excludes(staleNodes)`
is load-bearing for the entire reason classification, but no
test previously pinned the divergence. The closest existing
test, `TestSnapshot_IncludesAll`, counted 3 nodes in a snapshot
of 3 all-freshly-registered (healthy) nodes — it didn't
exercise the stale-inclusion path that feature 18 depends on.

## Test Gaps

### ~~T1 — `Registry.Snapshot()` vs `Registry.HealthyNodes()` stale-inclusion divergence unpinned~~ ✅ Fixed
**File:** `internal/cluster/registry_read.go:17-47`

Two methods on the same Registry, differing only in their
staleness filter:

```go
func (r *Registry) Snapshot() []*cpb.Node {
    // ... iterates r.nodes, emits all
}

func (r *Registry) HealthyNodes() []*cpb.Node {
    // ... iterates r.nodes, emits only e.isHealthy(r.staleAfter)
}
```

The divergence is the ENTIRE basis for `classifyUnschedulable`'s
ability to distinguish `all_matching_unhealthy` from
`no_matching_label`:

```go
func classifyUnschedulable(allNodes []*cpb.Node, selector map[string]string) string {
    // allNodes comes from Snapshot() — includes stale nodes
    for _, n := range allNodes {
        if nodeMatchesSelector(n, selector) {
            return events.UnschedulableReasonAllMatchingStale
        }
    }
    return events.UnschedulableReasonNoMatchingLabel
}
```

A regression that consolidated `Snapshot` and `HealthyNodes` —
e.g. a "cleanup" refactor that saw two near-identical methods
and merged them — would silently collapse the two reasons into
one. Every existing test would pass:

- `TestSnapshot_IncludesAll` asserts count only, and its 3
  nodes are all fresh-registered (healthy).
- `classifyUnschedulable` unit tests in
  `dispatch_unschedulable_reason_test.go` pass
  hand-crafted node slices directly, bypassing the Registry's
  filtering.
- The pass-09 integration test `TestDispatch_UnschedulableEvent_ReasonField_Correct`
  exercises the two reasons that don't require the stale-
  inclusion path (`no_healthy_node` via empty Registry,
  `no_matching_label` via healthy non-matching nodes).

Operator-visible regression: dashboards route stale-matches
to "add a new node" instructions when the real fix is to
restart the stale ones — misleading and wastes operator time.

**Fix:**
[`registry_test.go:TestSnapshot_IncludesUnhealthy_HealthyNodesExcludes`](../../../internal/cluster/registry_test.go)
registers two nodes ("fresh" and "stale"), sleeps past the
`staleAfter` window (2×testInterval + 10ms, mirroring existing
prune tests), then heartbeats only "fresh". Asserts:

- `Snapshot()` returns both nodes (len=2).
- The stale entry in the snapshot carries `Healthy=false`.
- `HealthyNodes()` returns only "fresh" (len=1).

A regression collapsing the two methods would fail either the
"snapshot must include stale" assertion or the
"HealthyNodes must exclude stale" assertion — both phrased so
the failure message names the feature-18 invariant being
broken.

Test is skipped in `-short` mode (mirrors the pattern of other
sleep-based staleness tests in the file).

## Calibration note

Pass 13 said feature 18 was likely saturated in one pass.
Pass 14 falsified that by drilling from the classification
wire-test (pass 09) back to the Registry invariant that feeds
it. The chain:

1. Dashboard consumes `reason` field → pass-09 wire test.
2. `reason` comes from `classifyUnschedulable(snapshot,
   selector)` → pass-09 unit test.
3. `classifyUnschedulable`'s `all_matching_unhealthy` path
   only fires when Snapshot includes stale nodes →
   **untested until T1**.
4. Snapshot's stale-inclusion is a Registry invariant, one
   line of code difference from HealthyNodes → now pinned.

Pattern update for dashboard-style features that consume
backend events: **trace each event field's origin back
through the source-of-truth data structure, then pin each
link in the chain**. Pass 13 pinned the HTTP layer (step 1);
pass 14 pinned the data-structure invariant (step 3). Neither
pass duplicated work.

Applied retroactively, this explains the feature-17 7-pass
marathon: each pass pinned a different link in the chain
(goroutine → mocked-contract → validator branches →
first-line defenses → metrics → persistence → cross-package
contract). The dashboard case is shallower — fewer links in
the chain because the dashboard is mostly a reader — but the
same drilling applies.

Pattern across features 11–18:

- 11: 4 passes
- 12: 4 passes
- 13: 2 passes
- 14: 1 pass
- 15: 1 pass
- 16: 3 passes
- 17: 7 passes
- 18: 2 passes (this one + pass 13)

Recommendation: feature 19 next. The iris demo is the
integration touchpoint for all prior features; its audit
should verify the demo actually exercises each feature's
critical path rather than trusting the demo script runs
end-to-end.
