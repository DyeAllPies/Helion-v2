**Overall Risk Level:** LOW.

Feature 11's production code is solid тАФ 47 unit tests plus
`TestS3_LiveIntegration` covered the surface from the first pass.
What this audit surfaced isn't wrong code, it's **missing alarms**:
invariants the feature doc explicitly depends on (root dir bits,
orphan-tempfile claim, mid-stream cancel semantics, WARN on
unencrypted S3) had no test that would fail if someone broke them.
Three sequential coverage passes during this audit closed all of
those alarms and wired the live-MinIO path into the e2e harness so
the existing live test actually runs in CI. No production-code
defects found.

## Medium

### ~~M1 тАФ Live-MinIO path was never exercised in CI~~ тЬЕ Fixed
**File:** `scripts/run-e2e.sh`, `docker-compose.e2e.yml`,
`internal/artifacts/s3_test.go:TestS3_LiveIntegration`

The artifact store's one live-S3 round-trip test was written to
skip unless `MINIO_TEST_ENDPOINT` was set, and **nothing in the
project ever set it**. The e2e harness (`scripts/run-e2e.sh`)
activated `COMPOSE_PROFILES=analytics` тАФ not `ml` тАФ so MinIO
wasn't even running. Net effect: every CI run, the only real-S3
test was silently skipped; a regression that broke minio-go's
real-endpoint interaction only surfaced in production.

**Fix:** (1) `scripts/run-e2e.sh` now activates
`analytics,ml` so MinIO + its bucket bootstrap come up alongside
the cluster. (2) `docker-compose.e2e.yml` sets
`HELION_ARTIFACTS_BACKEND=s3` + endpoint/bucket/creds on the
coordinator and both nodes, with `depends_on:
minio-bootstrap (service_completed_successfully)` so nothing
races bucket creation. (3) New "live-MinIO artifact tests" step
in the e2e script exports `MINIO_TEST_*` and runs both the
package-local live test and the new
`tests/integration/artifacts_live_s3_test.go` cases.

**Tests landed:** `TestS3_LiveIntegration` now runs on every
e2e pass; three new tests in `artifacts_live_s3_test.go`
(`TestLiveS3ArtifactRoundtrip`,
`TestLiveS3ArtifactVerifyStoreProbe` with happy + bad-bucket,
`TestLiveS3GetAndVerifyToStream` at 1 MiB).

### ~~M2 тАФ No unit test enforced the 0o700 root / 0o600 file mode bits~~ тЬЕ Fixed
**File:** `internal/artifacts/local.go:46-53`,
`internal/artifacts/local.go:73` (the explicit `os.MkdirAll(тАж, 0o700)`)

Feature 11 went out of its way to mode-restrict the artifact
root тАФ model checkpoints + raw training data are sensitive, and
`os.MkdirAll`'s default is 0o755. The code is correct, but no
test would notice if a future refactor silently dropped the
explicit mode arg. Security-relevant regression surface with no
alarm.

**Fix:** `local_test.go:TestLocalStore_Permissions` asserts the
root, a freshly-Put file, and a nested intermediate directory
all have exactly the documented bits. Skipped on Windows (NTFS'
mode-bit emulation isn't security-load-bearing anyway).

## Low

### ~~L1 тАФ The `UseSSL=false` WARN log had no test, a silent regression would hide a production misconfiguration~~ тЬЕ Fixed
**File:** `internal/artifacts/s3.go:109-114`

When an operator forgets `HELION_ARTIFACTS_S3_USE_SSL=1` in
production, the only signal is a WARN line that fires every time
an `S3Store` is constructed. If a future refactor silences or
conditionally gates that warning, operators lose the alarm on
unencrypted artifact traffic тАФ a harvest-now-decrypt-later risk
that `docs/SECURITY.md` ┬з 3 explicitly calls out. The message
was uncovered by any test.

**Fix:** `s3_test.go:TestNewS3Store_WarnsWhenUseSSLDisabled`
captures slog output via a JSON handler, asserts the WARN-level
record fires with the documented message and remediation hint,
and confirms the warning is absent when `UseSSL=true`.

### ~~L2 тАФ Doc-asserted invariant "no orphan tempfiles on concurrent same-key Put" had no test~~ тЬЕ Fixed
**File:** `internal/artifacts/local.go:82-108`

The feature 11 doc's "Deliberately not fixed" #2 explicitly
rebuts an earlier audit claim that racing Puts leak tempfiles,
arguing that `os.Rename` consumes the source and both racers'
tempfiles disappear. The argument is right but nothing encoded
it as a test.

**Fix:**
`local_test.go:TestConcurrentPuts_SameKey_NoOrphanTempfile`
spawns 16 goroutines, all Put the same key, and asserts:
(a) no errors, (b) exactly one resolved file remains, (c) no
`.helion-artifact-*.tmp` files survive, (d) the final bytes
equal one of the submitted payloads verbatim тАФ proving
rename is atomic, not a byte mix.

## Items considered and dismissed (not findings)

| Item | Why it's fine |
|------|--------------|
| `S3Store.Delete` TOCTOU between `StatObject` and `RemoveObject` | Doc-accepted behavior in feature 11's "Deliberately not fixed" #1. The race window is milliseconds wide, both observable outcomes are "object is gone," and no caller in the repo distinguishes "I deleted it" from "it was already gone." Writing a test that pins the accepted behavior would lock the design against a future fix тАФ not worth the flexibility cost. |
| `S3Store` endpoint format validation (accepting `http://` prefix as a typo) | minio-go silently accepts scheme-prefixed endpoints today and hits a confusing error on first request. Adding a validator at `NewS3Store` would be defensive overhead; the WARN log on `UseSSL=false` already gives operators a reading path if their production deploy went sideways. Leave to operational doc. |
| LocalStore `Stat` performance on multi-GB artifacts | `Stat` reads the whole file to recompute SHA-256 per call. The feature 11 spec explicitly says "Store is not in the coordinator's hot path." Benchmarks would be premature. |
| Library-caller bypass of API validators via `JobStore.Submit` directly | Doc-accepted in "Deliberately not fixed" #3 тАФ `safeJoin` re-validates every `local_path` at staging time. Two-layer defence is the right shape; no test added. |
| End-to-end node-agent тЖТ Stager тЖТ MinIO flow (upload on finalize, resolve on downstream job's input) | Not feature 11's surface тАФ that's feature 12 (Job I/O staging) and feature 13 (workflow artifact passing). Reviewed separately when those features get their coverage pass. |
