# helion-v2 Security & Code Quality Audit — 2026-04-15-08

**Audit ID:** 2026-04-15-08
**Date:** 2026-04-15
**Auditor:** Claude (Opus 4.6)
**Scope:** Feature 13 — ML workflow artifact passing. Second pass
after 2026-04-15-07 recommended moving on. Code re-read focused on
helpers that sit *around* the canonical resolver path — every place
the coordinator parses a `<upstream>.<output>` reference.
**Overall Risk Level:** LOW.

The first-pass audit declared feature 13 saturated after adding the
`ml.resolve_failed` event observer test. A second read-through found
a real divergence bug: two sibling helpers for parsing the same
`From` shape use **first-dot split**, while the canonical pair
(`api.SplitFromRef`, `cluster.splitDotRef`) uses **last-dot split**
per the documented contract. Workflow job names may contain dots
(no regex forbids them), so the bug is user-triggerable and
produces wrong diagnostics on the dashboard and wrong lineage
edges on the API. The `firstFromRef` source comment even claims
it "matches ResolveJobInputs's split rule" — a false claim that
would mislead a future reader into thinking the behaviour was
intentional. Calibration note 07 predicted a second pass would
find "contrived" gaps; this one found a documented-contract
violation instead. The recant stands.

## Findings

### ~~L1 — Two sibling `<upstream>.<output>` splitters used first-dot instead of last-dot~~ ✅ Fixed
**Files:**
- `internal/cluster/dispatch.go:78` (`firstFromRef`)
- `internal/cluster/workflow_lineage.go:215` (`splitFromRef`)

Feature 13's spec + API validator enforce that a `From` reference
splits at the **last** `.` so workflow job names containing dots
work. `api.SplitFromRef` and `cluster.splitDotRef` both honour
this — covered by `TestSplitFromRef` with the `{"a.b.c.D",
"a.b.c", "D"}` case.

But two sibling helpers, added later for the feature-18 event
emit and the workflow-lineage API, used `strings.IndexByte` /
hand-rolled first-dot loops. For a `From` value like
`model.v2.TRAIN` (upstream job `model.v2`, output `TRAIN`) they
would report upstream `model` / output `v2.TRAIN` — wrong in
both cases.

Visible impact:

- **`ml.resolve_failed` event** (dashboard Pipelines view): the
  event's `upstream` and `output_name` fields would carry the
  wrong values when the failing workflow used dotted job names.
  Operators looking at the dashboard to triage a broken pipeline
  would see a non-existent upstream job name and go hunting for
  the wrong thing.
- **Workflow lineage API** (`GET /api/workflows/{id}/lineage`):
  the returned `ArtifactEdges` would carry wrong
  `FromJob`/`FromOutput` values, rendering the wrong dependency
  graph on the dashboard's DAG view.

Neither is a data-corruption or security bug — the resolver
itself uses the canonical splitter, so the actual rewritten URI
is always correct. This is an observability/lineage bug.

The `firstFromRef` comment asserted "matches ResolveJobInputs's
split rule" — a false claim. A reviewer trusting the comment
would have bypassed a code-level check.

**Fix:** both helpers now use `strings.LastIndexByte` with the
same validation as `splitDotRef` (`dot > 0 && dot < len-1`).
Malformed refs (empty, leading dot, trailing dot) fall back to
the raw value as upstream for `firstFromRef` (operator
inspection — the event still fires with something) and return
`("", "")` for `splitFromRef` (the caller omits the edge).
Comments updated to reflect the correct contract and the reason
for the last-dot choice.

### ~~T1 — `firstFromRef` multi-dot case not pinned by any test~~ ✅ Fixed
**File:** `internal/cluster/dispatch_unschedulable_reason_test.go`

Existing `TestFirstFromRef` covered empty inputs, a single-dot
happy path, and the no-dot malformed case — but nothing asserted
the multi-dot case, so the first-dot bug above slipped past the
test suite. Added two cases: `model.v2.TRAIN` must split as
`("model.v2", "TRAIN")`, and trailing-dot `upstream.` falls back
to the raw value. The first locks the contract; the second pins
the malformed-ref fallback so a future tightening doesn't silently
change the event payload shape.

### ~~T2 — Workflow-lineage multi-dot ArtifactEdge not pinned~~ ✅ Fixed
**File:** `internal/cluster/workflow_lineage_test.go`

Existing tests covered the happy path (single-dot `a.b`
reference), malformed-empty, not-found, unstarted jobs,
model-store errors, and nil-models degradation — but no test
exercised the multi-dot case for lineage edges. Added
`TestBuildWorkflowLineage_MultiDotFromRef_LastDotSplit`: workflow
with job name `model.v2`, downstream input `from:
"model.v2.TRAIN"`, asserts the resulting `ArtifactEdge` carries
`FromJob="model.v2"` / `FromOutput="TRAIN"`.

## Items considered and dismissed (not findings)

| Item | Why not |
|------|---------|
| Duplicate output names on upstream `ResolvedOutputs` | `workflow_resolve.go:128-134` calls this out: first-wins if an upstream reports two outputs with the same `Name`. The feature-12 `attestOutputs` declared-name cross-check (fixed in 2026-04-15-05) already drops any reported output whose Name doesn't appear in `Job.Outputs`, and `firstDuplicateBindingName` in the submit validator rejects duplicate names at declaration time. Two entries with the same name cannot reach the resolver via legitimate paths; first-wins is the right conservative choice. |
| Workflow job name regex | There is no explicit regex on workflow job `Name` (only an emptiness check). A user could submit names with dots, slashes, or unicode. The last-dot split rule accommodates dots safely; slashes would collide with the `workflow_id/job_name` join used as Job ID. Not a feature-13 concern — belongs to a broader workflow validator pass. Noted and left. |
| Resolver failure event observer covers all three failure modes | 07's T1 covered `ErrResolveOutputMissing` (upstream completed without declared output). The other two (`ErrResolveUpstreamMissing` / `ErrResolveUpstreamNotCompleted`) are guarded by the dispatcher's eligibility gate and the workflow DAG validator — they are "shouldn't happen" paths. The event's payload fields don't vary by failure class, so adding two more observer tests would assert the same shape under contrived setup. Dismissed. |
| SHA-256 propagation correctness when upstream committed empty digest | Already covered implicitly — `findResolvedOutput` returns the pointer and the caller unconditionally copies `SHA256`. An empty digest propagates as empty, and the stager's verified-download path skips verification on empty — two already-covered invariants. A new test would just re-state what the existing happy-path test proves. |
| Resolver rejects a `From` ref with no dot | Shape is already enforced at three layers — `validateArtifactFromShape` rejects it at submit; `splitDotRef` returns `("","")` if it slips; the resolver surfaces `"from must be \"<upstream>.<output>\""`. Coverage across all three layers is sufficient; a resolver-only test for the no-dot case would duplicate the API validator's existing coverage. |
| `GetJobLineage` / `BuildWorkflowLineage` behaviour when the referenced upstream job is not yet scheduled | Already covered by `TestBuildWorkflowLineage_UnstartedJobs_PendingStatus`. |

## Calibration note

Audit 07 called feature 13 saturated after one pass. This
second-pass result argues against that call: one pass is not
enough for a feature whose surface touches four packages
(`api`, `cluster`, dashboard event consumer, lineage API). Pass
07's "one resolver, one DAG validator, one dispatcher site"
framing missed that feature 18 (dashboard) and feature 21
(workflow lineage API) both consume the same `<upstream>.<output>`
primitive — and each added its own helper rather than reusing
the canonical one, which is exactly the seam where drift creeps
in.

Revised rule of thumb: **one audit pass per primitive is
insufficient when sibling features consume the same contract
through their own helpers.** The consolidation fix would be to
lift `SplitFromRef` out of `internal/api` into a shared spot so
the `cluster`-package callers import one implementation instead
of re-rolling it. That's bigger than this audit and belongs to
a deferred cleanup. For now: four helpers that all agree on
last-dot split + tests that pin the contract at each site.

Recommendation: feature 14 next. If a future feature-13 third
pass is ever run, the question to ask is "does any new
helper-level consumer of the `From` primitive exist that wasn't
here when 07/08 were written?" — that's where the next
divergence would hide.
