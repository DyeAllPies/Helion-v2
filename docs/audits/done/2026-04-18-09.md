## Test Gaps

### ~~T1 â€” `/api/services/{job_id}` auth wrapper unpinned (401 path)~~ âœ… Fixed
**File:** `internal/api/handlers_services.go:34` â€” the
`s.authMiddleware(s.handleGetService)` wiring.

The route wrapper is the defense against unauthenticated
enumeration of running services. A refactor that accidentally
unwrapped the middleware (e.g. during handler-registration
cleanup) would make the endpoint public. Every existing test
uses `DisableAuth()` so the 401 path is exercised zero times.

Mirrors the existing admin-token 401 test at
`handlers_admin_test.go:249`.

**Fix:**
[`handlers_services_test.go:TestGetService_NoAuth_Returns401`](../../../internal/api/handlers_services_test.go)
uses `newAuthServer` (real token manager, auth enabled),
wires a fresh `ServiceRegistry`, sends `GET /api/services/any-job`
without an `Authorization` header, and asserts 401.

### ~~T2 â€” `/api/services` list-endpoint auth wrapper unpinned (401 path)~~ âœ… Fixed
**File:** `internal/api/handlers_services.go:33` â€” the
`s.authMiddleware(s.handleListServices)` wiring.

Same concern as T1 but on the list endpoint. A refactor could
plausibly drop the wrapper from one route while keeping it on
the other â€” the two are registered on adjacent lines by the
same function, so the alarm needs to fire per-route.

**Fix:**
[`handlers_services_test.go:TestListServices_NoAuth_Returns401`](../../../internal/api/handlers_services_test.go)
sends `GET /api/services` without auth and asserts 401.

### ~~T3 â€” `ReportServiceEvent` empty-JobId `InvalidArgument` branch unpinned~~ âœ… Fixed
**File:** `internal/grpcserver/handlers.go:490-492`

The first-line defense: empty JobId â†’ `codes.InvalidArgument`.
Without this branch:

1. Cross-node check: `jobs.Get("")` returns error, condition
   short-circuits (no violation recorded).
2. Registry upsert: the registry's own empty-JobID guard at
   `service_registry.go:48` silently skips.
3. `LogServiceEvent` fires regardless â€” an audit row with
   empty `job_id` and no corresponding registry state.
4. Client receives `Ack{Ok: true}` and proceeds, unaware its
   event was dropped.

Dropping the `InvalidArgument` branch would convert loud
failure into silent data loss with audit-log noise.

**Fix:**
[`report_service_event_test.go:TestReportServiceEvent_EmptyJobId_Rejected`](../../../internal/grpcserver/report_service_event_test.go)
sends a `ServiceEvent{JobId: ""}` and asserts
`codes.InvalidArgument`. Mirrors the existing cross-node-mismatch
rejection test shape (which uses `codes.PermissionDenied`).

## Items considered and dismissed (not findings)

| Item | Why not |
|------|---------|
| `SetServiceRegistry(nil)` vs. `SetServiceRegistry` never called â€” semantic difference | The doc comment at `handlers_services.go:30` implies passing nil skips route registration; the code always registers routes and relies on the nil-check in the handlers. Both paths return 404 to the client â€” a purely internal distinction with no regression target. The existing `TestGetService_NotFoundWhenRegistryUnwired` covers the "never called" case. |
| Empty-JobId check at `handlers_services.go:84` in the HTTP handler | `r.PathValue("job_id")` on route `GET /api/services/{job_id}` never returns empty â€” the Go 1.22+ mux requires the path variable to match. The fallback `if jobID == "" { 400 }` branch is defensive code that can never fire given the routing. Testing would exercise net/http.ServeMux routing. |
| Prober grace-period startup delay (`HealthInitialMs > 0` branch) | The `if spec.HealthInitialMs > 0 { select { ... } }` at `service_prober.go:59-65` wraps `time.After`. Testing requires wall-clock observation â€” flaky. The prior pass-06 tests use `HealthInitialMs: 0` specifically to avoid time-dependency. A regression dropping the branch would cause probes to fire immediately on startup rather than waiting; operational impact is a few premature-unhealthy events, not a data-integrity break. |
| Goroutine-leak test for `probeService` after ctx cancel | The goroutine exits on `ctx.Done()` â€” Go's `select` semantics guarantee this. Testing requires goroutine-counting (`runtime.NumGoroutine`) which is inherently flaky under concurrent test execution. The prober's ctx-done path is exercised implicitly by every pass-06 test (the `done := make(chan struct{})` channel blocks on goroutine exit). |
| `GRPCNodeDispatcher.DispatchToNode` `ack.Accepted == false` error formatting | The error message format (`"node %s rejected job %s: %s"`) is operator-facing but not load-bearing for any programmatic consumer. A regression changing the format string would fail human-facing log grep, not machine-facing behaviour. |
| `ServiceRegistry.Snapshot()` mutation safety â€” modifying the returned slice should not affect the registry | Go slice semantics: the returned slice shares the backing array only if the caller never re-slices or overwrites past `len(out)`. The registry's snapshot builds a fresh `make([]cpb.ServiceEndpoint, 0, len)` + appends, so mutations to the returned slice do not bleed into the registry. Testing would pin stdlib guarantees. |
| `ReportServiceEvent` with `evt.Ready == false` (unhealthy state) via the RPC harness | The handler branches on `evt.Ready` only for the slog event-name string (`service.ready` vs `service.unhealthy`). Upsert + audit paths are identical. The pass-06 integration test's flip sub-test already exercises Ready=false end-to-end via the prober, so a ready-false RPC test would duplicate coverage under different setup. |
| `ReportServiceEvent` rate limiting | Unlike the registry's `registryQueryAllow`, the feature-17 spec explicitly does not rate-limit ReportServiceEvent â€” edge-triggering bounds the volume (one RPC per readiness flip per job). A test pinning "no rate limit" would alarm for a design change, not a regression. |
| IPv4 node address with port not stripped when building UpstreamURL | Handled by `net.SplitHostPort`; covered by pass-07's IPv6 test + existing IPv4 happy path. Edge cases like unbracketed IPv6 (`::1:9090`) would fall through SplitHostPort's error path, which pass-07's dismissed items log covers. |

Each evaluated against the audit framework bar â€” **would a real
