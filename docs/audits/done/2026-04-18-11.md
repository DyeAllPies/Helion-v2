metrics. This pass tested that rule by checking a surface pass
10 didn't mention: **BadgerDB persistence**.
**Overall Risk Level:** LOW.

Pass 10 declared saturation at five consumer surfaces. This
sixth pass found a missed producer surface: `cpb.Job.Service`
is persisted to BadgerDB via `SaveJob` and re-hydrated on
coordinator restart via `LoadAllJobs`. The round-trip is
load-bearing for crash recovery — without it, a coordinator
restart mid-inference would lose the Service spec on every
pending service job, and each would recover as a batch job
(no prober, no readiness event, `/api/services/{id}` stays
404 forever).

Peer features with similar persisted fields have dedicated
round-trip + forward-compat tests:
- Feature 14: `persistence_labels_test.go:TestBadgerPersister_NodeLabels_Roundtrip`
  + `ForwardCompatFromPreLabelJSON`
- Feature 15: `TestBadgerPersister_NodeTotalGpus_Roundtrip` +
  `ForwardCompatFromPreGPUJSON`
- Feature 17: **missing**

Calibration: pass 10 listed four **consumer** surfaces but didn't
call out **producer** surfaces. Persistence is a distinct
producer-surface class — the coordinator writes to BadgerDB to
survive restart, then reads back through the same serialisation
path. A JSON-tag typo in the struct definition silently breaks
the restart contract without touching any consumer.

## Test Gaps

### ~~T1 — `Job.Service` BadgerDB Save→LoadAll round-trip unpinned~~ ✅ Fixed
**File:** `internal/cluster/persistence_jobs.go:21` — the
`json.Marshal(j)` call that serialises `cpb.Job` including its
`Service *ServiceSpec` field.

The existing `TestBadgerPersister_SaveJob_LoadAllJobs_Roundtrip`
at `persistence_test.go:116` only inspects `ID` + `Command`. A
regression changing the JSON tag on `cpb.Job.Service` (`json:"service,omitempty"`)
or on any of `ServiceSpec`'s three fields (`port` / `health_path`
/ `health_initial_ms`) would silently strip those fields from
the persisted row. On restart:

1. Coordinator crashes with a pending service job.
2. `LoadAllJobs` reads the row but the unmarshal drops the
   broken tags → `Service = nil`.
3. Recovery dispatches the job; `node_dispatcher.go:78` sees
   `job.Service == nil` and skips the `pb.ServiceSpec` wiring.
4. Node treats it as a batch job, runs `python -m myserver`
   to completion.
5. No prober fires, no `ReportServiceEvent` RPC, no registry
   entry. Client calls `GET /api/services/{id}` forever and
   always gets 404.

The existing per-feature tests for labels and TotalGpus pinned
this exact regression class for their respective fields. Service
was missed.

**Fix:** new
[`persistence_service_test.go:TestBadgerPersister_JobService_Roundtrip`](../../../internal/cluster/persistence_service_test.go)
saves a Job with a populated ServiceSpec (port, health path,
grace period), loads via `LoadAllJobs`, asserts all three
fields survive the JSON round-trip.

### ~~T2 — Forward-compat from pre-feature-17 Job JSON unpinned~~ ✅ Fixed
**File:** `internal/cluster/persistence_jobs.go:45` — the
`json.Unmarshal(v, &j)` call that materialises the Job back.

The `omitempty` tag on `Service` means a pre-feature-17
batch-job row (no service field) serialises with no `"service"`
key at all — Go's `json.Unmarshal` defaults `Service` to nil
when the key is missing. This invariant is load-bearing for
mixed-version deployments: a new coordinator reading an old
batch-job row must not crash the unmarshal or synthesise a
phantom non-nil ServiceSpec.

Symmetric to `TestBadgerPersister_NodeLabels_ForwardCompatFromPreLabelJSON`
and `TestBadgerPersister_NodeTotalGpus_ForwardCompatFromPreGPUJSON`.
A future refactor (e.g., dropping `omitempty` or switching
field types) could break this without tripping any other test.

**Fix:**
[`persistence_service_test.go:TestBadgerPersister_JobService_ForwardCompatFromPreServiceJSON`](../../../internal/cluster/persistence_service_test.go)
saves a Job without Service, loads, asserts
`jobs[0].Service == nil`.

## Items considered and dismissed (not findings)

| Item | Why not |
|------|---------|
| `omitempty` behaviour — Service=nil should emit no `"service"` key | `encoding/json` guarantees omitempty elides zero-value fields. The forward-compat test (T2) implicitly exercises this path (saves without Service, loads, asserts nil). A dedicated JSON-bytes inspection would pin stdlib behaviour, not Helion code. |
| `ServiceSpec` value vs. pointer copy semantics | The field is `*ServiceSpec`; Go's JSON marshaling handles pointer-to-struct the same as the underlying struct. Testing pointer semantics would alarm on a Go-version change, not a Helion regression. |
| `Job.Service` persistence with concurrent SaveJob | BadgerDB's `Update` transaction serialises per-key writes by construction. Concurrent-save correctness is a BadgerDB guarantee; testing it would alarm on a stdlib regression. |
| Workflow persistence of Service-bearing jobs (WorkflowJob.Service?) | Workflow jobs use `cpb.WorkflowJob` which is a distinct type. At the time of audit, `WorkflowJob` does not carry a ServiceSpec field — services are standalone-submit only. If that changes (service-as-workflow-step), a new persistence test belongs in workflow persistence, not here. |
| Downgrade compat — an old coordinator reading a new Service-bearing row | Old versions would fail to unmarshal unknown fields gracefully (Go's default is to skip). Not a tested pattern elsewhere in Helion — downgrade is not a supported migration direction. |
| BadgerDB TTL on service-job rows | Job entries have no TTL per `persistence_jobs.go:18-19`. A regression adding TTL would be caught by every other job-related persistence test. |
| `cpb.ServiceEndpoint` (registry) persistence | Deliberately NOT persisted per `service_registry.go:13-19` ("In-memory only. A coordinator restart starts with an empty map; the nodes' probers re-emit the current state"). Testing "ServiceEndpoint isn't persisted" would alarm on a design change. |
| `ReportServiceEvent`'s `OccurredAt` roundtrip through `timestamppb` | Wire-format concern handled by protobuf-generated code. The registry Upsert honours `OccurredAt` via the existing `TestServiceRegistry_Upsert_KeepsCallerTime`. |

Each evaluated against the audit framework bar — **would a real
regression actually trip this test?** and **is the alarm
qualitatively different from what's already in place?** Only
T1 and T2 cleared both.

## Calibration note

Pass 10 said feature 17 had four consumer surfaces and declared
saturation across them. Pass 11 found a fifth surface — the
**persistence producer surface**. Pattern update:

Consumer surfaces (what reads feature state):
1. HTTP clients
2. gRPC peers
3. Audit-log queries
4. Metrics scrapers

**Producer surfaces** (what writes feature state):
5. Persistence (BadgerDB via JSON)
6. Events bus (covered via observer tests during consumer-side audits)

The distinction matters because pass 10's consumer-enumeration
lens naturally missed the persistence write path. Producer
surfaces share a common regression class: the feature's data
must survive restart / re-hydration through a serialisation
pipeline that no consumer test exercises.

Generalised rule: **for every persisted field, the feature's
audit must include a roundtrip test AND a forward-compat test
against a row missing that field**. Features 14 and 15 did this
for labels and TotalGpus. Feature 17 missed it for Service
until pass 11.

Feature 17 is now saturated across six surfaces. A seventh pass
would need to find a surface I haven't thought of — at this
point genuinely contrived. The feature's audit pass count (6)
reflects its distinctive breadth: long-running goroutine +
cross-layer wiring + five independent downstream surfaces.

Pattern across features 11–17:

- 11: 4 passes
- 12: 4 passes
- 13: 2 passes
- 14: 1 pass
- 15: 1 pass
- 16: 3 passes
- 17: 6 passes

Recommendation: feature 18 (dashboard) next. The "consumer +
producer" framing updated here applies; feature 18 is primarily
a consumer surface for events pinned in features 12–17, but
it has its own producer surface (audit-log writes from UI
actions) worth enumerating.
