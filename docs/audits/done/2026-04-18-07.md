## Test Gaps

### ~~T1 — `audit.Logger.LogServiceEvent` has no focused event-type + details test~~ ✅ Fixed
**File:** `internal/audit/logger.go:187-205`

The method is the only entry in the audit package's logger
surface without a matching `TestLog*_EventType` / `Details` test
in `logger_test.go`. The shape being pinned — there is no
alternative:

- **Event type string flips on readiness**: `"service.ready"` /
  `"service.unhealthy"`. These strings are the primary filter
  key for audit queries (`Query.Type`). A rename would silently
  break every existing query.
- **Actor format `"node:"+nodeID`**: the `"node:"` prefix groups
  events by reporting node in SIEM queries. Every other actor
  in the audit surface uses a prefix convention (`"user:"`,
  `"system"`, `"unknown"`). A regression dropping the prefix
  would mix service-event actors with user-actor events of the
  same literal string.
- **Details map keys**: `job_id`, `port`, `health_path`,
  `consecutive_failures`. Each one is consumed by at least one
  downstream path (dashboard lineage → job_id; operational
  alerts → consecutive_failures; debug log grep → port +
  health_path).

Pass 06's integration test at
`internal/nodeserver/service_prober_test.go` verifies the call
*happens* (via the mock's `serviceEvents` counter) but the mock
ignores all arguments. The real logger's shape is untested.

**Fix:**
[`logger_test.go:TestLogServiceEvent_Ready_Unhealthy`](../../../internal/audit/logger_test.go)
with two sub-tests. Each registers a real `audit.Logger` with a
mock store, invokes `LogServiceEvent`, then inspects the stored
event's Type, Actor, and every field in Details. `ready=true`
produces `"service.ready"`; `ready=false` with
`consecutive_failures=3` produces `"service.unhealthy"` with the
correct count (pins the counter propagation the dashboard's
alerting query depends on). JSON decoding returns numeric
details as `float64` — pinned explicitly so a type change would
surface.

### ~~T2 — `buildUpstreamURL` IPv6 branch untested~~ ✅ Fixed
**File:** `internal/api/handlers_services.go:117-127`

The function has an explicit IPv6 path that wraps the host in
square brackets for URL construction (`http://[::1]:8080/path`).
Without the brackets the output is ambiguous to URL parsers and
every HTTP client rejects it. The IPv4 test
(`TestGetService_LiveEndpoint_Returns200`) covers the common
deployment, but any operator running a dual-stack or IPv6-only
cluster hits the untested branch.

A regression dropping the `net.ParseIP(host).To4() == nil`
check — or the bracket-wrap inside — would produce malformed
URLs in production, silently visible only to IPv6 operators.

**Fix:**
[`handlers_services_test.go:TestGetService_IPv6NodeAddress_WrapsBrackets`](../../../internal/api/handlers_services_test.go)
inserts a `cpb.ServiceEndpoint` with `NodeAddress="[::1]:9090"`,
hits `GET /api/services/{id}`, and asserts `UpstreamURL ==
"http://[::1]:8080/healthz"`. A single test for the one branch
the IPv4 test didn't exercise.

## Items considered and dismissed (not findings)

| Item | Why not |
|------|---------|
| `buildUpstreamURL` includes `HealthPath` in the returned upstream URL (`/healthz` suffix) | The existing test explicitly pins `http://10.0.0.4:8080/healthz` as expected output at `handlers_services_test.go:42`. This is a deliberate design choice, not a bug — clients using the returned URL for a health poke get the full path wired, while clients wanting the base URL can construct it from the separate `NodeAddress` + `Port` fields in the same response. Changing it would be a spec-level discussion, not an audit finding. |
| Probe `StatusCode >= 200 && < 300` — treatment of 3xx redirects | The HTTP client in `probeService` has no `CheckRedirect` override, so Go's default would follow up to 10 redirects. But a health endpoint returning 3xx is misconfigured by definition — the probe would follow, and the final status determines ready/unhealthy. Testing "3xx treated as unhealthy" would exercise the Go HTTP client, not Helion code. |
| 4 KiB body drain cap in prober | Documented at `service_prober.go:87-91`: "Cap the drain at 4 KiB so a misbehaving /healthz that streams megabytes of body cannot slow the prober down." Testable via an httptest server that streams, but the regression class is "prober hangs on a slow service" — a timing-dependent test that would add flakiness without catching the class of bug the cap prevents. The cap itself is `io.CopyN(io.Discard, resp.Body, 4<<10)`, a standard-library guarantee. |
| `advertiseAddr` propagation from SetAdvertiseAddress → Dispatch → ServiceEvent → Registry → UpstreamURL | End-to-end propagation is covered by the pass-06 integration test (the registry entry's NodeAddress is what drives the upstream URL). A standalone test for the accessor would pin getter/setter behaviour. |
| `ReportServiceEvent` with empty `evt.NodeAddress` | The coordinator's `ServiceEndpoint.NodeAddress` field is stamped from the prober's `s.advertiseAddr()` which is stamped once at node startup. Empty is a misconfiguration; the resulting UpstreamURL would have an empty host (`http://:8080/healthz`) which clients reject. Same "operator misconfiguration surfaces immediately" shape as the IPv6 case — but the IPv6 case has an explicit code branch to pin; the empty-address case is fallthrough code already exercised by every test. |
| `ServiceRegistry.Snapshot()` / `Count()` concurrent modification | `sync.RWMutex` guarantees the whole package. Testing concurrent Read/Write would exercise stdlib, not Helion. |
| Feature 17's `IsService` cross-cut with `Jobs` page on the dashboard | Deferred to feature 18 audit — the consumer side is tested there. |

Each evaluated against the audit framework bar — **would a real
regression actually trip this test?** and **is the alarm
qualitatively different from what's already in place?** Only
T1 / T2 cleared both.

## Calibration note

Pass 07 found two gaps that pass 06's integration-test lens
missed by construction: pass 06 used a mock audit logger (since
its focus was the prober goroutine), which meant the real
`audit.Logger.LogServiceEvent` implementation stayed untested.
Generalised rule: **when a pass uses mocks to isolate the
component under test, always check whether the mocked component
has its own focused test elsewhere.** If not, the mocked
contract is load-bearing but unpinned.

This is the mirror image of pass 06's rule ("long-running
goroutines need multi-tick integration tests"). Together they
form a two-sided guarantee: the caller is tested end-to-end AND
the callee is tested in isolation.

Pattern across features 11–17:

- 11: 4 passes → cross-backend contract saturation
- 12: 4 passes → multiple production fixes
- 13: 2 passes → splitter divergence
- 14: 1 pass → event payload observer
- 15: 1 pass → API boundary
- 16: 3 passes → security + validator + audit-log observer
- 17: 2 passes → prober + audit/IPv6 residuals

Recommendation holds: feature 18 (dashboard ML module) next.
