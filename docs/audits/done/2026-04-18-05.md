independent `s.audit.Log` call at each register/delete handler.
**Overall Risk Level:** LOW.

Every registry write handler emits TWO independent side-effects:
a bus event (via `s.eventBus.Publish`) and an audit log entry
(via `s.audit.Log`). Audit 03 pinned the bus-event side for
`model.registered` and implicitly for delete/dataset variants.
The audit-log side was entirely unpinned ÔÇö no test wires an
`audit.Logger` into a registry server and asserts the entries
land. A refactor that extracted the audit-call pattern into a
helper and forgot to invoke it from one handler, or renamed
`dataset.registered` to a new convention, would leave every
existing registry test green while silently breaking the
compliance / audit-query surface.

One integration test covering all four emissions (register +
delete ├Ś datasets + models) is enough ÔÇö they share the same
`s.audit.Log` pattern, so a refactor breaking one is likely to
break all of them. The alarm proves the pattern holds end-to-end.

## Test Gaps

### ~~T1 ÔÇö `s.audit.Log` emission path unpinned for every registry handler~~ Ôťů Fixed
**Files:**
- `internal/api/handlers_registry.go:122` (`dataset.registered`)
- `:210` (`dataset.deleted`)
- `:280` (`model.registered`)
- `:391` (`model.deleted`)

Each handler wraps the audit call in `if s.audit != nil`; none
are tested end-to-end with a real `audit.Logger`. The
`cluster`, `admin`, and `analytics` test surfaces all wire an
`inMemoryAuditStore` + `audit.NewLogger` to prove the emission
path. Registry did not.

Why this matters as a distinct alarm from the event-bus
observer fixed in audit 03:

- The two side-effects travel different code paths. The event
  bus uses `s.eventBus.Publish` returning void; the audit log
  uses `s.audit.Log` returning error (logged via
  `logAuditErr`). Dropping one doesn't imply dropping the other.
- Downstream consumers differ. The event bus feeds the
  dashboard's realtime view; the audit log feeds
  `GET /audit` history queries and the SIEM sink. Losing the
  audit log silently breaks compliance, not observability.
- The project memory note on `always_tests_and_hybrid_security`
  treats audit emission as a first-class concern ÔÇö every
  privileged mutating endpoint must ride an audit trail or a
  regression slips past the project's security posture.

**Fix:**
[`handlers_registry_test.go:TestRegistry_AuditLog_RegisterAndDelete_BothResources`](../../../internal/api/handlers_registry_test.go)
+ new `newRegistryServerWithAudit` helper (mirrors
`newAnalyticsServerWithAudit`). The test performs
register + delete on one dataset and one model (four handler
calls, four expected audit entries), scans the in-memory audit
store, and asserts every expected `event_type` appears with the
correct actor ("anonymous" since auth is disabled in the test
server). Four emissions covered in one test; a refactor
breaking any single emission trips the "audit entry missing"
assertion for that topic.

## Items considered and dismissed (not findings)

| Item | Why not |
|------|---------|
| Symmetric amendments to existing event-bus tests to pin `actor` + `uri` on `dataset.registered` | The `TestRegistry_Dataset_RegisterPublishesEvent` asserts name/version/size_bytes but not actor/uri. Audit 03 dismissed this as "CreatedBy roundtrip is tested elsewhere implicitly" ÔÇö a defensible call at the time. Now that the audit-log side has its own observer, the actor field is pinned on one of the two emission paths for dataset-register, which is sufficient: a refactor dropping actor from both paths would trip the new audit-log test; a refactor dropping actor from only the bus-event payload would still leave the audit trail intact. Amending the bus-event test would add a second alarm for a single failure mode. |
| Audit log for `model.registered` omits `source_dataset` (event bus includes it) | Already dismissed in audit 04: deliberate spec-level split documented in the feature-16 doc's audit event taxonomy table. Not a regression target. |
| Audit entry key collision under high write rates (`audit:<nanos>:<seq>:<id>` format) | `audit.Logger` uses an atomic seq counter specifically to break same-nanosecond ties (logger.go:102-103). The collision case is structurally impossible. Testing would pin a property guaranteed by a lock-free primitive. |
| `logAuditErr` behaviour when the underlying audit store returns an error | The handler logs the error via `logAuditErr(false, ...)` and proceeds ÔÇö the user-visible operation (register/delete) still succeeds, matching the design principle "audit failure must not block the operation." Testing this would require a store that fails on Put, which `failOnScanStore` already covers for the Scan direction. The success-with-failed-audit path exercises a tiny error-handling branch that the existing `logAuditErr` tests cover at the audit-package layer. |
| TTL behaviour on audit entries | The handler passes `ttl=0` via `audit.NewLogger(store, 0)` which disables expiry. Audit entries live forever. A regression changing this would show up in the `audit.Logger` tests at the audit-package layer, not the registry consumer. |
| Race between register-then-delete within a single `s.audit.Log` round-trip | The Log call is synchronous; there's no async pipeline to race. Any regression is a code-structural bug caught by static analysis, not runtime race detection. |

Each evaluated against the audit framework ÔÇö **would a real
regression actually trip this test?** and **is the alarm
qualitatively different from what's already in place?** Only
T1 cleared both bars.

## Calibration note

Third feature-16 pass converged on a distinct alarm class that
the first two passes didn't surface: the independent audit-log
emission path. Pass 03 found three surface-level security /
observability gaps. Pass 04 found one validator behaviour bug.
Pass 05 found the audit/event asymmetry ÔÇö both emissions come
from the same handler but travel different interfaces.

The pattern is now stable enough to generalise into an audit
playbook entry:

**For every REST handler that emits side-effects on success:**
1. Pin the HTTP contract (status codes, body shape, error
   translation) Ôćĺ surface pass
2. Pin each validator boundary and line-level logic Ôćĺ
   validator pass
3. Pin each side-effect emission independently (event bus,
   audit log, metrics counter, external webhook) Ôćĺ emission
   pass

Feature 16 took three passes to walk these three phases. The
pattern should apply to every feature with a REST surface;
features 11ÔÇô15 (non-REST-heavy) passed in 1ÔÇô2 passes each.

Recommendation: feature 18 next per audit 03's standing
recommendation. It's the primary consumer of the event-bus
payloads pinned across audits 07, 09, and 03 ÔÇö its audit will
test the other side of those contracts.
