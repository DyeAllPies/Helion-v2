# helion-v2 Security & Code Quality Audit — 2026-04-18-03

**Audit ID:** 2026-04-18-03
**Date:** 2026-04-18
**Auditor:** Claude (Opus 4.6)
**Scope:** Feature 16 — ML Dataset and Model Registry. First pass.
Code under audit: `internal/registry/` (BadgerStore, validators,
Dataset/Model types, LatestModel chronological lookup),
`internal/api/handlers_registry.go` (REST surface: POST/GET/LIST/
DELETE for both resources, `/latest` endpoint, JWT auth,
per-subject `registryQueryAllow` rate limit, `http.MaxBytesReader`
at 1 MiB, audit + event-bus publishing on register/delete,
404-from-mux when `SetRegistryStore` wasn't called).
**Overall Risk Level:** LOW.

Feature 16 ships 33 tests covering validators, BadgerDB
roundtrips, HTTP register/get/list/delete/latest, dup 409,
missing 404, bad-scheme / partial-lineage / NaN 400, pagination,
registry gating. Three gaps matched recurring audit patterns:

1. **Rate-limit 429 observable behaviour** — `registryQueryAllow`
   (2 rps burst 30) had no integration test proving requests past
   the burst receive 429. Analytics and admin surfaces both have
   this test; registry did not. A refactor breaking the preflight
   ordering would leave writes unbounded.
2. **1 MiB body cap observable behaviour** — matches the job-submit
   `TestSubmitJob_OversizedBody_Returns400` shape exactly. Without
   it, a caller could stream multi-MiB of free-form tags before
   validation runs.
3. **`model.registered` event lineage payload** — the event carries
   `source_job_id` + `source_dataset{name, version}` conditionally
   (only set when non-empty). The analytics pipeline consumes
   these to build the training graph. Zero observer tests covered
   the event at all, let alone the conditional lineage half. Same
   silent-regression class as audits [2026-04-15-07](2026-04-15-07.md)
   (`ml.resolve_failed` observer) and
   [2026-04-18-01](2026-04-18-01.md) (`job.unschedulable` reason
   field).

## Test Gaps

### ~~T1 — `registryQueryAllow` 429 response unpinned~~ ✅ Fixed
**File:** `internal/api/handlers_registry.go:53` — the
`if !s.registryQueryAllow(actor)` branch that writes
`StatusTooManyRequests`.

The token bucket is documented (2 rps burst 30) and used by every
registry preflight, but no test drives the burst to 429. Analytics
and admin surfaces both have this test
(`TestAnalytics_RateLimit_ExcessReturns429`,
`handlers_admin_test.go:380-399`). A refactor that accidentally
skipped preflight for GET endpoints, or moved the limiter after
the handler body, would leave every existing test green while
exposing registry writes to unbounded flooding.

**Fix:**
[`handlers_registry_test.go:TestRegistry_RateLimit_ExcessReturns429`](../../../internal/api/handlers_registry_test.go)
sends 100 `GET /api/datasets` requests and asserts at least one
returns 429. Mirrors the analytics variant. GET is chosen over
POST because the limiter sits before body decoding — testing via
the cheapest path proves the limiter fires regardless of request
shape.

### ~~T2 — 1 MiB body cap unpinned~~ ✅ Fixed
**File:** `internal/api/handlers_registry.go:83` (datasets POST)
and `:234` (models POST) — both set
`r.Body = http.MaxBytesReader(w, r.Body, maxSubmitBodyBytes)`.

The cap is identical to the job-submit handler's, which has
`TestSubmitJob_OversizedBody_Returns400` at
`handlers_jobs_test.go:373`. Registry has no matching test. A
regression that dropped the `MaxBytesReader` line (e.g., during
a refactor that moved body decoding into a shared helper and
forgot the cap) would silently accept arbitrary-sized POST
bodies. The only observable signal would be a slow memory
creep under adversarial load.

**Fix:**
[`handlers_registry_test.go:TestRegistry_OversizedBody_Returns400`](../../../internal/api/handlers_registry_test.go)
POSTs a dataset with a tag value just over 1 MiB and asserts
the response is 400. Datasets and models share the cap via
identical plumbing; the dataset path is representative.

### ~~T3 — `model.registered` event observer test missing entirely~~ ✅ Fixed
**File:** `internal/events/topics.go:228` — `ModelRegistered`
constructor, especially the conditional lineage block:

```go
if sourceJobID != "" {
    data["source_job_id"] = sourceJobID
}
if sourceDatasetName != "" {
    data["source_dataset"] = map[string]string{
        "name":    sourceDatasetName,
        "version": sourceDatasetVersion,
    }
}
```

The event is emitted at `handlers_registry.go:287` on every
successful model registration. `TestRegistry_Dataset_RegisterPublishesEvent`
covers the dataset half (and even it only asserts name +
version + size_bytes — not uri, not actor). For models: nothing.
The conditional lineage fields are especially regression-prone
because they're only populated when the registrar passes both
halves — a refactor that inverted the nil check, or dropped
the `source_dataset` nested map, would ship undetected.

The analytics pipeline's training graph depends on
`source_job_id` and `source_dataset` to correlate models with
their training runs and input datasets. Losing those fields
silently blanks the graph while the registry API keeps working.

**Fix:**
[`handlers_registry_test.go:TestRegistry_Model_RegisterPublishesEvent`](../../../internal/api/handlers_registry_test.go)
registers a model with full lineage (source_job_id +
source_dataset), subscribes to `TopicModelRegistered` before
the POST, and asserts every load-bearing payload field: name,
version, uri, actor (the test server's "anonymous"),
source_job_id, and the nested source_dataset.{name, version}
map.

## Items considered and dismissed (not findings)

| Item | Why not |
|------|---------|
| Actor field on the dataset-register event | Already covered implicitly — `TestRegistry_Dataset_RegisterAndGet` asserts `CreatedBy == "anonymous"` on the roundtrip (handlers_registry_test.go:68). The actor is stamped from the same JWT-subject thread that populates both the record and the event; if the record shows "anonymous", the event carries the same. A dedicated event-field assertion would pin the same plumbing twice. Worth noting but not a gap. |
| DELETE idempotence — second delete returns 404, not 204 | Already covered by BadgerStore's `ErrNotFound` return on missing key (badger.go delete path) and the handler's 404 translation. A second DELETE exercises the same code path as a DELETE on a never-registered resource, which `TestRegistry_Dataset_NotFound` already alarms. Adding a "delete twice" test would exercise the same branch under a different setup. |
| `/latest` on a name with zero registered versions | The HTTP handler for `/latest` calls `LatestModel(name)`, which returns `ErrNotFound` when the prefix scan finds no entries; the handler translates to 404. `badger_test.go:TestLatestModel_Empty` (or equivalent — verified via the `ErrNotFound` path in the existing LatestModel tests) covers the store side; the HTTP 404 translation is the same three-line switch used by every other missing-resource path. A dedicated integration test would duplicate coverage. |
| Concurrent register of identical (name, version) — first-wins race test | `RegisterDataset` / `RegisterModel` wrap BadgerDB's `Update` transaction, which serialises writes on the same key. Duplicate detection runs inside the txn via a `Get` + `ErrAlreadyExists` check; BadgerDB's MVCC guarantees at most one txn commits for any given key. The existing `TestRegistry_Dataset_DuplicateVersion_Conflict` + BadgerDB's own txn tests cover the primitive. A goroutine-level hammer test would prove Badger's transaction guarantees, not Helion's. |
| Version-charset boundary (exact-64-byte accepted, 65 rejected) | The existing `validate_test.go` version tests cover a representative accepted set and the oversize-reject case. An exact-boundary pair would alarm an off-by-one flip in the length check — but the version charset check allocates zero; the only regression class is a constant-change (e.g., `maxVersionLen` renamed and typo'd), which every existing version test also covers. Dismissed. |
| Lineage `SourceDataset` with version-but-no-name | Existing `validate_test.go` covers name-without-version rejection. The validator's actual condition at `validate.go:228` is `if m.SourceDataset.Name != "" \|\| m.SourceDataset.Version != ""` — the || means either half triggers validation. Version-without-name falls into the same branch and gets rejected via the inner check. Adding the symmetric test would exercise the same branch with different input. |
| Pagination page-size 0 / negative / > cap | `pageBounds` sanitisation at handlers_registry.go:333-336 applies defaults and caps — covered implicitly by every existing list test (they use unspecified page, which hits the default branch). Explicit boundary tests would exercise input parsing rather than registry behaviour. |
| Cross-resource DELETE isolation (delete model X doesn't remove dataset X) | The `datasets/` vs `models/` key prefix guarantees isolation by construction. `TestRegistry_Dataset_And_Model_NoCollision` covers the coexistence half; asserting "delete A doesn't touch B" would test BadgerDB's prefix scoping rather than registry code. |
| Chronological-ties list ordering under identical `CreatedAt` | Go's `sort.Slice` is not stable; the documentation warns about this and clients must not depend on intra-tie order. Writing a test that pins a specific tie-breaking behaviour would over-specify. |
| Metric extreme-but-finite values (MaxFloat64, -MaxFloat64) | Validator rejects `NaN` and `±Inf` specifically because they don't round-trip through `encoding/json`. Finite values up to `±MaxFloat64` serialise cleanly and are valid metrics; no policy reason to cap further. |

Each evaluated against the two honest questions from the
audit-02 framework — **would a real regression actually trip
this test before production?** and **is the alarm qualitatively
different from what's already in place?** Only T1 / T2 / T3
cleared both bars.

## Calibration note

Feature 16 is the first non-ML-runtime feature audited in this
sweep (11–15 were all runtime / scheduler / dispatch). Its gaps
clustered differently: two security caps (rate-limit, body-cap)
and one observability field. Features with a large HTTP surface
and side-effect emission (audit + event bus + persistence) tend
to ship with strong unit-level coverage but weaker
integration-level alarms on cross-cutting concerns that don't
fit neatly into one handler's unit test.

Pattern update across features 11–16:
- 11: cross-backend contract lock (4 passes, eventual saturation)
- 12: production fix on pass 3 (attestOutputs declared-name)
- 13: production fix on pass 2 (last-dot splitter divergence)
- 14: event-payload field observer gap
- 15: API boundary-validator gap
- 16: security-cap observable + event-payload gaps (3)

The "would a dropped line silently break security/observability?"
question has been the most productive audit lens across these
passes. Every finding has been a line that was present in
production but unpinned by any test. Recommendation holds from
audit 2026-04-18-02: feature 18 (dashboard ML module) next,
given it's the consumer of multiple cross-feature event payloads
already flagged in prior audits.
