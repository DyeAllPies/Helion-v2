## Test Gaps

### ~~T1 — No test proved `LocalStore` and `S3Store` behave identically under the same call sequence~~ ✅ Fixed
**File:** `internal/artifacts/contract_test.go` (new)

Each backend's own test file verifies that backend is internally
consistent — but nothing compared them head-to-head under one
sequence of operations. A future refactor of either backend could
drift its error values (e.g. swapping `ErrNotFound` for a wrapped
`fmt.Errorf`), its handling of empty payloads, or its
double-delete semantics, and every per-backend suite would still
pass because each is self-consistent. The divergence would only
surface when the Stager — which doesn't care which backend it
talks to — started behaving subtly differently against one
backend, and that's exactly the kind of regression integration
tests are bad at catching cleanly.

**Fix:**
[`TestStoreContract_IdenticalAcrossBackends`](../../../internal/artifacts/contract_test.go)
parametrises over both backends and runs one sequence — Put
(unique key), Get, Stat, Delete, Get-after-delete, Stat-after-
delete, second Delete, Put (empty payload), Get (empty), Stat
(empty) — asserting each step produces the same observable
result. Runs in ~10 ms; proves `errors.Is(err, ErrNotFound)`
works identically, proves both backends accept empty payloads,
pins the well-known empty-string SHA-256 constant as a shared
invariant.

The live-MinIO backend isn't parametrised here (it requires
network + skip logic) — it's covered by its own integration tests
in [`tests/integration/artifacts_live_s3_test.go`](../../../tests/integration/artifacts_live_s3_test.go).
The two in-process backends (`LocalStore` + `S3Store` via
`fakeS3`) are what differ in implementation and so what matters
to pin.

## Coverage saturation

After four coverage passes totalling **7 added tests across unit,
integration, and e2e layers**, I'm calling further coverage
diminishing returns. Everything I considered adding in this pass
and chose not to:

| Considered | Why not |
|------------|---------|
| Boundary test: key at exactly `MaxKeyLength` (1024) | Existing `TestPut_OversizeKeyRejected` covers over-limit; at-limit is a one-off boundary that the sanitiser's `>` (not `>=`) arithmetic already handles correctly-by-construction. Writing this test would pin a boundary that has never regressed. |
| Symlink-as-store-root behaviour | The LocalStore resolves via `filepath.Abs` + explicit join; symlinks would be dereferenced. No production path creates a symlinked store root. Speculative. |
| S3 region-mismatch (bucket in us-east-1, config says us-west-2) | minio-go's error on region mismatch is its own contract, not ours. The WARN log already makes operator misconfig visible in the startup signal. |
| Disk-full / ENOSPC for `LocalStore.Put` | Hard to simulate hermetically without a loopback filesystem; the Put's error path is exercised by `TestPut_SizeMismatch_LeavesNoFile` which covers the "partial write, clean up" shape. |
| Multi-node concurrent Put against live MinIO | That's S3's last-writer-wins contract, not ours. We don't mediate. |
| Partial download recovery | `GetAndVerifyTo` is strict — it does not resume partial downloads and the doc doesn't promise it will. Testing the absence of a feature isn't useful. |
| Registering the same dataset twice via the UI returns 409 | Real test value but it's registry-layer (feature 16) behaviour, not artifact-store. The registry's own unit + handler tests already cover 409. |
| Performance benchmarks (throughput, latency percentiles) | Not a correctness concern. `Stat` reads the whole file on LocalStore to compute SHA-256 — that's an O(n) cost documented in the feature spec as acceptable for the registry's non-hot-path usage. |

Each of these was evaluated against two honest questions: **"would a
real regression actually trip this test before production?"** and
**"is the alarm this would set qualitatively different from what's
already in place?"** None cleared both bars.

The suite now has:

- **50+ unit tests** — per-backend coverage + contract lock + permission
  bits + WARN-log observability + concurrent-same-key invariant + ctx-
  mid-stream cancellation.
- **4 integration tests** against live MinIO — round-trip, VerifyStore
  probe (happy + bad-bucket), 1 MiB stream, 10 MiB round-trip.
- **1 Playwright spec** — register/list/filter/delete via dashboard
  against `HELION_ARTIFACTS_BACKEND=s3`.

That's the full coverage picture. Subsequent audits should focus on
the *downstream consumers* (Stager, registry, workflow resolver) or
on new code — not on this Store.
