# helion-v2 Security & Code Quality Audit — 2026-04-15-05

**Audit ID:** 2026-04-15-05
**Date:** 2026-04-15
**Auditor:** Claude (Opus 4.6)
**Scope:** Feature 12 — ML job I/O staging. Third coverage pass after
[`2026-04-15-04`](2026-04-15-04.md) declared saturation. Re-audit
specifically to find whether the saturation claim was right.
**Overall Risk Level:** LOW (defence-in-depth gap in the
coordinator's attestation trust boundary; not exploitable for
cross-job data exfiltration but observable as misleading
dashboard / audit-log entries).

The "saturation" declaration in the prior audit turned out to be
premature. One genuine production-code gap was found in the
coordinator's `attestOutputs` trust boundary: the function checks
prefix + scheme + shape but **does not cross-reference the
reported output Names against the Job's submit-time `Outputs`
declaration**. A compromised node controls bytes under its own
`jobs/<id>/` prefix; under the previous behaviour it could also
register those bytes under any Name it pleased, polluting the
operator's view of the job and the audit trail. Now fixed at the
source — `attestOutputs` takes a `declaredNames` set and drops any
reported entry whose Name isn't in it.

## Low

### ~~L1 — `attestOutputs` accepted reported outputs whose Name was not in `Job.Outputs`~~ ✅ Fixed
**File:** `internal/grpcserver/handlers.go` —
`(*Server).attestOutputs` and its call site in `ReportResult`.

The coordinator's documented "trust boundary for node-attested
artifact metadata" enforced shape + scheme + per-job key prefix.
What it did **not** enforce was that a reported output's `Name`
matched something the Job's submit-time `Outputs` declaration
asked for. A compromised node has full control over its own
`jobs/<id>/` key space (it has the bytes); the prefix check above
already proves the URI lives there. What was unchecked was the
follow-on step where the coordinator records the entry under
whatever `Name` the node attached, exposing it to:

- The Pipelines / Models dashboard view, which reads
  `Job.ResolvedOutputs` verbatim — operators would see fabricated
  output names attributed to legitimate jobs.
- The audit trail's `job.completed` event, which carries the
  outputs list.
- `Job.ResolvedOutputs`-walking code paths in feature 13's
  workflow resolver. The DAG validator at submit time blocks
  legitimate downstream `from: this-job.<undeclared>` references,
  so the data-exfiltration path is closed by belt-and-braces;
  this audit closes the suspenders.

Severity is Low because the cross-job data-exfiltration path is
already blocked by the DAG validator (a downstream `from:` that
references an undeclared upstream Name fails at submit time). The
remaining damage was operator-confusion + audit-trail integrity,
both real but bounded.

**Fix:**
- `attestOutputs` now takes a `declaredNames map[string]struct{}`
  argument and drops any reported entry whose `Name` isn't in
  the set. nil-safe — passing nil disables the check, which the
  pre-existing unit tests in
  [`security_test.go`](../../../internal/grpcserver/security_test.go)
  use to keep their focus on the prefix-mismatch path.
- `ReportResult` builds the set from `job.Outputs` at the call
  site, so the cross-check fires automatically on every real
  RPC.
- A dropped entry logs a Warn line and emits a
  `LogSecurityViolation` audit event with reason
  `output_name_undeclared` — same shape as the existing
  `output_prefix_mismatch` violation so the operational alarm
  surface is consistent.

**Tests landed (T1 + T2 below).**

## Test Gaps

### ~~T1 — Compromised-node smuggling an undeclared output Name had no test~~ ✅ Fixed
**File:**
[`report_result_test.go:TestReportResult_AttestsOutputs_DropsUndeclaredName`](../../../internal/grpcserver/report_result_test.go)

Job declares only `MODEL`. Node reports `MODEL` + `EVIL`, both
with valid prefixes and shapes. Test asserts `MODEL` is the only
entry in the persisted `ResolvedOutputs`; the WARN line for the
dropped `EVIL` confirms the new code path fires.

### ~~T2 — Job with no declared Outputs at all but node reports some had no test~~ ✅ Fixed
**File:**
[`report_result_test.go:TestReportResult_AttestsOutputs_NoOutputsDeclared_DropsAll`](../../../internal/grpcserver/report_result_test.go)

A batch job with empty `Outputs` shouldn't have any
ResolvedOutputs after attestation, even if the node reports
some. The empty declared-set passed in by the handler causes
attestOutputs to drop everything — locks in the fail-closed
default behaviour.

Both tests pass; the existing
`TestReportResult_AttestsOutputs_DropsForgedEntries` test from
[`2026-04-15-03`](2026-04-15-03.md) was updated to declare its
LEGIT output, since the new check would have dropped it
otherwise. That update doesn't weaken the original test — it now
asserts BOTH the prefix-mismatch drop (FORGED's URI names a
different job) AND that the new declared-name check lets the
LEGIT output through.

## Items considered and dismissed (not findings)

| Item | Why not |
|------|---------|
| Cross-check `LocalPath` against the declaration too | The declared `LocalPath` is what the stager mints the URI from, but the coordinator can't cheaply re-derive it on the read side without re-running the validator. The Name check covers the practical attack; LocalPath consistency follows from the URI prefix check (which already enforces `jobs/<id>/<localpath>`). |
| Fail the whole RPC instead of dropping silently | The existing pattern across `attestOutputs` is to drop + log + audit + still complete the job's transition. Failing the RPC would let a compromised node hold the coordinator's job-completion path hostage by reporting bad outputs. |
| Audit-log batching for repeated drops on the same job | Each drop is one audit row. A node that reports 64 undeclared outputs would write 64 rows. Bounded; not worth a coalescing layer. |
| Dashboard alert on undeclared-name drops | Belongs to feature 18's audit, not feature 12's — the surface that surfaces audit events is the Events view. |
| Backward compatibility shim (accept undeclared outputs from "old" nodes) | Helion-v2 has no v1 nodes in the wild and no compatibility window to honor. The break-the-bad-path-now choice is right. |

## Note on the prior audit's saturation declaration

[`2026-04-15-04`](2026-04-15-04.md) declared feature 12 coverage
saturated. That declaration was wrong — this audit found a real
production-code defence-in-depth gap on the third look. The
right response isn't to apologise for the false signal; it's to
update the saturation language with calibration. The pattern that
actually matters: each successive audit pass should find
qualitatively smaller things, and at some point the next pass
finds nothing genuinely new. This pass found something. The next
one might or might not.

For feature 12, this audit's findings exhaust what I can see at
the source-reading level. Whether further passes find more is an
empirical question, not a declarable fact.
