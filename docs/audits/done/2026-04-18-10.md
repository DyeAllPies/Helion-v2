## Test Gaps

### ~~T1 ‚Äî `helion_services_total` Prometheus gauge emission unpinned~~ ‚úÖ Fixed
**File:** `internal/metrics/prometheus.go:218-221` ‚Äî the
`if c.services != nil { ch <- prometheus.MustNewConstMetric(
c.descServicesTotal, ...) }` block in `Collect()`.

The gauge is load-bearing for operational dashboards that
track live inference-service count across the cluster. The
wiring chain is:

1. `cluster.ServiceRegistry.Count()` satisfies
   `metrics.ServiceCounter`.
2. `metrics.NewRegistry(..., serviceCounter)` calls
   `coll.SetServiceCounter(s)`.
3. `Collect()` emits the gauge on scrape when
   `c.services != nil`.

The only reference to the gauge in test code is a comment
counting descriptors in `TestCollector_Describe_SendsDescriptors`
(`internal/metrics/prometheus_test.go:45`). No test asserts:

- The gauge appears in scrape output when a `ServiceCounter`
  is wired.
- The gauge value matches `Count()`.
- The `SetServiceCounter` path flows from `NewRegistry`'s 5th
  parameter through to emission.

A regression dropping any link in the chain would silently
remove the gauge from scrape output ‚Äî the operational
dashboard would just show a flat line or "no data" with no
error signal.

**Fix:**
[`prometheus_test.go:TestNewRegistry_ServicesGauge_Emitted`](../../../internal/metrics/prometheus_test.go)
adds a `mockServiceCounter` mirroring `mockRegistryCounter`.
Wires `NewRegistry(..., nil, sc)` where `sc.n = 4`, scrapes
`/metrics`, asserts the body contains `"helion_services_total 4"`.
Pins both the emission AND the value propagation.

### ~~T2 ‚Äî `helion_services_total` omitted when counter is nil unpinned~~ ‚úÖ Fixed
**File:** `internal/metrics/prometheus.go:219` ‚Äî the
`if c.services != nil` guard.

Complementary alarm to T1. A coordinator that never opts into
feature 17 (no `SetServiceCounter` call, or `NewRegistry(...,
nil, nil)`) must not emit the gauge ‚Äî otherwise non-ML
deployments would see a misleading `helion_services_total 0`
line on scrape that could alarm operators who aren't running
inference at all.

Symmetric to `TestNewRegistry_RegistryGauges_OmittedWhenNilCounter`
at line 179.

**Fix:**
[`prometheus_test.go:TestNewRegistry_ServicesGauge_OmittedWhenNilCounter`](../../../internal/metrics/prometheus_test.go)
calls `NewRegistry(..., nil, nil)`, scrapes `/metrics`,
asserts the body does NOT contain `"helion_services_total"`.

## Items considered and dismissed (not findings)

| Item | Why not |
|------|---------|
| Descriptor count in `TestCollector_Describe_SendsDescriptors` should pin the services descriptor explicitly | The existing test asserts `count > 0` rather than a specific number ‚Äî adding a specific-descriptor assertion would duplicate the Collect-side observer test (T1) which already implicitly proves the descriptor is both registered and emitted. |
| `ServiceCounter.Count()` returning a pathological value (negative, MaxInt64) | `Count()` returns `len(r.endpoints)` which is non-negative and bounded by process memory. Testing `len() < 0` would alarm on a violation of Go runtime guarantees. |
| Prometheus metric type for `helion_services_total` (GaugeValue vs CounterValue) | `GaugeValue` is correct (services count goes up and down). The existing dataset/model gauges use the same pattern; a wrong-type regression would fail Prometheus registration via panic at `MustRegister`, which any test that calls NewRegistry already catches. |
| Label fidelity on `helion_services_total` (no labels per the descriptor) | The descriptor passes `[]string(nil)` for labels ‚Äî this is a stable API choice. Adding labels would be a feature addition, not a regression target. |
| `ServiceCounter` interface method set | The interface is `Count() int`, a single-method contract. Go structural typing auto-verifies ‚Äî no test can assert the interface has exactly one method. |
| `NewRegistry`'s position-5 ordering of the `serviceCounter` parameter | Argument order is a source-level API concern; a regression reordering would fail to compile, not run. |
| `Describe` count matching `Collect` count exactly | Prometheus requires the two be consistent; violations panic at registration. No test needed at this layer. |
| Gauge emission under concurrent `Count()` mutation | `ServiceRegistry.Count()` uses `sync.RWMutex`; a Prometheus scrape mid-Upsert gets either the pre- or post-Upsert count. Both are valid. No linearisation test can distinguish. |

Each evaluated against the audit framework bar ‚Äî **would a real
regression actually trip this test?** and **is the alarm
qualitatively different from what's already in place?** Only
T1 and T2 cleared both.

## Calibration note

Pass 09 said "a fifth pass would need to invent gaps." Pass
10 falsified that with two non-invented gaps in the metrics
surface. The error: pass 09 treated feature 17 as a
coordinator ‚Üî node wire feature + HTTP read surface, but
missed that feature 17 also **owns an operational metric** via
its dedicated `ServiceCounter` interface. Metrics is a distinct
consumer surface from HTTP / gRPC, and the pattern-search lens
I used in pass 09 didn't cover it.

Updated rule: **when auditing a feature, enumerate every
consumer surface** ‚Äî HTTP clients, gRPC peers, event-bus
subscribers, audit-log queries, **and metrics scrapers**. Each
is a distinct contract worth pinning. Feature 17's four
consumer surfaces:

1. HTTP clients (`GET /api/services/{id}`, `GET /api/services`)
   ‚Äî pinned across passes 06‚Äì09.
2. gRPC peers (`ReportServiceEvent`) ‚Äî pinned across passes
   06, 09.
3. Audit-log consumers (`LogServiceEvent` rows) ‚Äî pinned in
   passes 06, 07.
4. **Metrics scrapers (`helion_services_total`)** ‚Äî pinned in
   pass 10.

Feature 17 is now saturated across all four surfaces. A sixth
pass would genuinely find contrived items; at five passes this
feature is tied with 11 and 12 for highest pass count and has
the most diverse gap classes (goroutine, mocked-contract,
validator branches, first-line defense, metrics).

Pattern across features 11‚Äì17:

- 11: 4 passes
- 12: 4 passes
- 13: 2 passes
- 14: 1 pass
- 15: 1 pass
- 16: 3 passes
- 17: 5 passes

Recommendation: feature 18 (dashboard ML module) next. It
consumes the events pinned across feature-17 passes; the
dashboard audit closes the read side of every event-bus
contract.
