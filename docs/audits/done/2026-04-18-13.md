## Test Gaps

### ~~T1 — `GET /workflows/{id}/lineage` authMiddleware wrapper unpinned~~ ✅ Fixed
**File:** `internal/api/server.go:167` — the
`s.mux.HandleFunc("GET /workflows/{id}/lineage",
s.authMiddleware(s.handleGetWorkflowLineage))` wiring.

The lineage response carries workflow topology, all job IDs,
and pointers to every registered model produced by each job.
Unauthenticated access would disclose the cluster's full ML
lineage graph. Every peer workflow handler
(`POST /workflows`, `GET /workflows/{id}`, `DELETE
/workflows/{id}`) is wrapped in the same middleware but the
lineage wrapper specifically has no 401 test. A refactor that
moved the registration out of `SetWorkflowStore` into a
different helper and forgot to re-wrap would slip past every
existing test.

Matches the pattern pinned for the feature-17 services
endpoints in audit [2026-04-18-09](2026-04-18-09.md)'s T1/T2
and for admin endpoints in `handlers_admin_test.go:249`.

**Fix:**
[`handlers_workflows_lineage_test.go:TestGetWorkflowLineage_NoAuth_Returns401`](../../../internal/api/handlers_workflows_lineage_test.go)
uses `newAuthServer` (real token manager), wires the workflow
store, sends `GET /workflows/any-id/lineage` without an
`Authorization` header, and asserts 401.

### ~~T2 — `ErrWorkflowLineageNotFound` → 404 translation unpinned at HTTP layer~~ ✅ Fixed
**File:** `internal/api/handlers_workflows_lineage.go:44-47`

```go
if errors.Is(err, cluster.ErrWorkflowLineageNotFound) {
    writeError(w, http.StatusNotFound, "workflow not found")
    return
}
```

The library test `TestBuildWorkflowLineage_NotFound` in
`internal/cluster/workflow_lineage_test.go` verifies the
library returns the sentinel; the handler's translation to
404 is a separate concern. A refactor that swapped the
error-translation switch (e.g. a new `default: 500` clause
placed above the specific `errors.Is` check) would silently
convert "workflow doesn't exist" into "internal server
error" — misleading for clients who need to distinguish
"retry later" (500) from "give up, typo'd the ID" (404).

**Fix:**
[`handlers_workflows_lineage_test.go:TestGetWorkflowLineage_UnknownWorkflow_Returns404`](../../../internal/api/handlers_workflows_lineage_test.go)
drives the handler with a fully-wired store but a bogus
workflow ID; asserts 404 and that the response body is not an
"internal error" string.

### ~~T3 — Happy-path JSON response shape unpinned at HTTP layer~~ ✅ Fixed
**File:** `internal/api/handlers_workflows_lineage.go:55-56`
— the `writeJSON(w, "handleGetWorkflowLineage", lineage)`
emission.

The library tests verify `BuildWorkflowLineage` returns a
correctly-shaped `*WorkflowLineage`; the handler's JSON
encoding to the wire is a separate step. A regression that
changed `writeJSON`'s content-type header, dropped the
`application/json` declaration, or altered the JSON field
names via a struct-tag change would silently break the
dashboard's client-side decoder while every library test
stayed green.

**Fix:**
[`handlers_workflows_lineage_test.go:TestGetWorkflowLineage_Success_Returns200`](../../../internal/api/handlers_workflows_lineage_test.go)
submits a minimal two-job workflow (`build` → `test`),
GETs `/workflows/lineage-happy/lineage`, asserts 200, JSON-
decodes the response, and spot-checks that `workflow_id`,
`jobs` length, and the dependency edge (`test.depends_on
["build"]`) survive the handler-level encoding. Doesn't
duplicate field-level lineage assertions covered at the
library layer — just proves the HTTP→JSON pipe works.

## Items considered and dismissed (not findings)

| Item | Why not |
|------|---------|
| 404 when `SetWorkflowStore` was never called (routes unregistered) | Symmetric to the `SetServiceRegistry` unwired path that pass 09 tested: the mux returns 404 from its own handler when the route isn't registered. The handler's internal `if s.workflowStore == nil` guard at line 27 is unreachable via HTTP because SetWorkflowStore is what registers the route. Testing it would exercise ServeMux's 404 handler, not our code. |
| Graceful degrade when `s.models == nil` (coordinator without registry) | Documented behaviour at handlers_workflows_lineage.go:31-38: the handler sets `modelReader = nil` and library-level `BuildWorkflowLineage` serves the lineage with empty `models_produced` slices. Covered by `TestBuildWorkflowLineage_NilModels_DegradesGracefully` at the library layer; the HTTP handler is a pass-through on the nil case. |
| Generic 500 translation for non-sentinel errors from BuildWorkflowLineage | The handler has a `default` switch that wraps unknown errors into 500 via `writeError(..., http.StatusInternalServerError, "internal error")`. A test would need a library-level error injection which would require either a fake LineageWorkflowReader that errors or a struct refactor. The existing library tests cover `TestBuildWorkflowLineage_PropagatesModelStoreError`; the HTTP-layer 500 translation is one line and symmetric to every other handler in the package. |
| Dashboard-side `UnschedulableReason*` constants validation | The dashboard doesn't currently render these as user-friendly labels (the Pipelines event-rollup is explicitly deferred to `deferred/26-pipelines-event-integration.md`). When it does, a dashboard-side test pinning the string-to-label map belongs in Angular code; for now the events flow through `/events` as raw strings. |
| `GET /api/services` list pagination | The endpoint doesn't paginate by design — the spec notes "memory-only read on the coordinator side; no rate limit needed." A pagination test would alarm on a non-existent feature. Covered-as-designed by the existing `TestListServices_*` tests which verify the full-list return. |
| `ServiceRegistry.Snapshot()` concurrent-mutation safety | `sync.RWMutex`-guarded; testing concurrent access pins a stdlib guarantee. |
| `ml.resolve_failed` event observer on the dashboard | Feature 18's spec explicitly lists this as an "emitter → surfaced on /events and audit log"; the producer-side observer test landed in audit [2026-04-15-07](2026-04-15-07.md). Dashboard-side consumption of the event is deferred to backlog entry 26. |
| mermaid render error handling in the Pipelines DAG detail view | Dashboard concern; Angular test file `ml-pipeline-detail.component.spec.ts` explicitly notes the render path is async + DOM-heavy and out of unit-test scope. A regression would surface on manual QA or docker e2e. |
| Register-dataset dialog `parseTags` edge cases | Dashboard concern; `register-dataset-dialog.component.spec.ts` tests the normal path. Edge-case coverage (trailing comma, whitespace) is minor UI polish; the authoritative tag validation happens server-side via `registry.ValidateTags` which has its own tests. |

Each evaluated against the audit framework bar — **would a real
regression actually trip this test?** and **is the alarm
qualitatively different from what's already in place?** Only
T1, T2, T3 cleared both.

the backend side — most of its producer code landed inside
other features' slices and got audited there. The one surface
unique to feature 18 is the Pipelines DAG view's lineage

feature audit finds a focused set of gaps. Don't expect the
