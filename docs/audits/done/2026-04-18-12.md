## Test Gaps

### ~~T1 — `GoRuntime.Run` IsService no-timeout branch untested~~ ✅ Fixed
**File:** `internal/runtime/go_runtime.go:128-130`

```go
if req.IsService {
    timeout = 0
}
```

This branch overrides whatever `TimeoutSeconds` the dispatcher
forwarded (via `cluster.NodeDispatcher` → `pb.DispatchRequest`
→ `runtime.RunRequest`). Without it, a service job with
`TimeoutSeconds: 60` (or the default 5-minute timeout) gets
killed after that interval — directly contradicting the
feature-17 spec: "The job is treated as long-running (no
timeout enforcement, terminal state is only reached on
explicit stop or process exit)."

A regression dropping the branch would:

1. Submit a service job (e.g., `python -m myserver`).
2. Runtime applies `defaultTimeout` (5 min).
3. Prober fires first `ready` event, registry populates,
   GET /api/services/{id} returns 200.
4. 5 minutes later: runtime kills the subprocess.
5. Dispatch returns, Job transitions to terminal, registry
   entry deleted via JobCompletionCallback.
6. Client's next request gets 404 — the service "silently died."

No existing test catches it because:
- `TestGoRuntime_Timeout` tests `TimeoutSeconds=1` without
  `IsService` (timeout fires).
- `TestGoRuntime_DefaultTimeout_Is5Minutes` checks the field
  value, not the IsService interaction.
- All runtime tests use plain batch-job RunRequests with no
  `IsService: true`.

Pass 06's dismissal claim — "tested at runtime layer" — was
factually incorrect.

**Fix:**
[`go_runtime_test.go:TestGoRuntime_IsService_IgnoresTimeout`](../../../internal/runtime/go_runtime_test.go)
starts a `sleepCmd` with `IsService: true` and
`TimeoutSeconds: 1`. Observation pattern:

- goroutine: `rt.Run(ctx, req)`; report `done <- res`.
- Main: `select { <-done: fail("terminated early"); <-time.After(1500ms): pass }`.
- If the branch is broken: timeout=1s fires at 1 s,
  goroutine returns within 1.5 s → test fails with a clear
  "IsService did not disable timeout" message.
- If the branch works: process runs past 1.5 s → test cancels
  ctx and asserts `KillReason != "Timeout"`.

The 1.5 s observation window gives 50% margin over the 1 s
TimeoutSeconds it should ignore — comfortably outside timer
scheduling noise on Windows/Linux CI.

## Calibration note

Pass 11 said feature 17 was saturated across consumer +
producer surfaces. Pass 12 found a gap via a different axis:
**cross-package contract verification**. Feature 17's
implementation spans `internal/api` (validator / handlers),
`internal/cluster` (service registry, node dispatcher),
`internal/nodeserver` (prober), `internal/grpcserver` (RPC
handler), `internal/audit` (logger method), `internal/metrics`
(gauge), `internal/runtime` (IsService handling), and
`internal/proto/coordinatorpb` + `/proto` (types). Each
package has its own test file; each package's contract for
feature 17 needs direct coverage.

Pass 06's dismissal of the IsService test was an assumption
that peer packages covered what they didn't. This class of
miss is distinct from surface enumeration — it's about
checking actual test coverage per package, not just reasoning
about what "should" be covered.

Generalised rule: **for every feature that spans N packages,
grep each package's test files for the feature's specific
identifiers (field names, constants, function names) and
verify each is referenced.** A zero-reference identifier in a
test directory is a concrete, searchable marker of an untested
contract.

Applied retroactively: pass 07 already used this pattern to
find `LogServiceEvent` in audit/logger.go was absent from
audit/logger_test.go. Pass 12 used the same pattern on
runtime/go_runtime.go's `IsService`. The heuristic works.

Feature 17 is now saturated across consumer surfaces + producer
surfaces + cross-package contract verification. An eighth pass
would need a genuinely new framing axis. At seven passes,
feature 17 is the most-audited feature in the codebase;
justified by its exceptionally wide surface area (eight
packages, six consumer/producer surfaces, one cross-package
contract).

Pattern across features 11–17:

- 11: 4 passes
- 12: 4 passes
- 13: 2 passes
- 14: 1 pass
- 15: 1 pass
- 16: 3 passes
- 17: 7 passes

Recommendation: feature 18 (dashboard) next. The
cross-package-grep rule applied here transfers directly —
dashboard code lives in `dashboard/` (Angular) + `internal/api`
(server-sent events? REST?) + potentially other places.
