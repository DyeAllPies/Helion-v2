round-trip for the Service field — the two submit-side surfaces

Found four branch-level gaps:
- 3 unpinned validator branches (oversize health path,
  whitespace in path, oversize grace period) — each is a
  distinct `return` path in `validateServiceSpec` that no test
  exercises. Removing any one of these checks would leave every
  existing validator test green while silently accepting invalid
  input.
- 1 unpinned API response field (`JobResponse.Service` — the
  service block a client sees via `GET /api/jobs/{id}`).
  `TestSubmit_Service_Accepted` covers the submit status code
  but never fetches the job back. A refactor dropping the
  `if j.Service != nil { resp.Service = ... }` block at
  helpers.go:67-73 would silently strip service metadata from
  every GET response while submit kept working.

None are critical individually; together they close the submit
→ validator → response triangle for feature 17.

## Test Gaps

### ~~T1 — `validateServiceSpec` oversize-health-path branch unpinned~~ ✅ Fixed
**File:** `internal/api/handlers_jobs.go:273-275`

The validator caps health paths at `maxHealthPathLen = 256`
bytes. Existing tests cover empty and relative paths; no test
exercises the length cap. A regression dropping the branch
would accept arbitrarily long paths that land on the prober's
URL string — potentially megabytes.

**Fix:**
[`handlers_services_validation_test.go:TestSubmit_Service_RejectsOversizeHealthPath`](../../../internal/api/handlers_services_validation_test.go)
submits `/` + 257 `a` bytes and asserts 400.

### ~~T2 — `validateServiceSpec` whitespace/NUL health-path branch unpinned~~ ✅ Fixed
**File:** `internal/api/handlers_jobs.go:276-278`

Defense-in-depth charset check — `strings.ContainsAny(s.HealthPath,
" \t\r\n\x00")`. Primary concern: CRLF injection into the probe
URL string. Go's `net/http` catches this at request-construction
time too, but the submit-time check is first-line defense.
Removing it would accept `/path\r\nHost: evil` paths; downstream
sanitisation might leave a gap.

**Fix:**
[`handlers_services_validation_test.go:TestSubmit_Service_RejectsHealthPathWithWhitespace`](../../../internal/api/handlers_services_validation_test.go)
submits `/has space` and asserts 400. All four rejected byte
classes (space, tab, CR, LF, NUL) share the same `ContainsAny`
check so one representative input suffices.

### ~~T3 — `validateServiceSpec` oversize-grace-period branch unpinned~~ ✅ Fixed
**File:** `internal/api/handlers_jobs.go:279-281`

`HealthInitialMs` capped at `maxServiceHealthInitialMs` (30 min
= 1_800_000 ms). Operational-correctness check: a misconfigured
grace of 24 hours would delay failure detection far beyond any
sane limit. The regression class is a dropped upper-bound check
— values like 2_000_000 would silently pass.

**Fix:**
[`handlers_services_validation_test.go:TestSubmit_Service_RejectsOversizeGracePeriod`](../../../internal/api/handlers_services_validation_test.go)
submits `health_initial_ms: 2000000` (clearly over the 30 min
cap) and asserts 400.

### ~~T4 — `JobResponse.Service` round-trip unpinned~~ ✅ Fixed
**File:** `internal/api/helpers.go:67-73`

`jobToResponse` copies `j.Service` onto the outgoing
`JobResponse.Service`. The `TestSubmit_Service_Accepted` test
exercises the submit path but never GETs the job back. A
refactor dropping the `if j.Service != nil { ... }` block —
e.g. during an unrelated helpers.go cleanup — would silently
strip the service block from every GET response. Clients
relying on `GET /api/jobs/{id}.service` to discover the probed
port would see `null` with no error signal.

**Fix:**
[`handlers_services_validation_test.go:TestSubmit_Service_Response_RoundtripsServiceSpec`](../../../internal/api/handlers_services_validation_test.go)
POSTs a job with service `{port: 9090, health_path: "/ready",
health_initial_ms: 500}`, then GETs `/jobs/svc-roundtrip` and
asserts every field lands in the response unchanged.

## Items considered and dismissed (not findings)

| Item | Why not |
|------|---------|
| Dispatcher → node `ServiceSpec` forwarding at `node_dispatcher.go:78-84` | A regression dropping the `req.Service = &pb.ServiceSpec{...}` block would cause service jobs to dispatch without the spec — no prober runs, no events fire, `/api/services/{id}` stays 404. **But** this is end-to-end observable: any docker e2e test that submits an inference job and GETs `/api/services/{id}` would fail loudly. Docker e2e is planned for after features 11–19 are audited (user-stated plan). Testing it now via a real gRPC harness (cluster → node wire) would duplicate what docker e2e gives for free in a few days. |
| `GRPCNodeDispatcher.DispatchToNode` over a real gRPC listener | Same as above — the RPC surface is where unit tests stop being cheap. The dispatcher's build-and-dial function is not easily factorable without a prod-level refactor (extract `buildDispatchRequest(job)`), and the alarm value is indistinguishable from what e2e will catch. |
| `JobCompletionCallback` triggering `serviceRegistry.Delete` | Callback wiring lives in `cmd/helion-coordinator/main.go` + existing workflow-lifecycle tests. The delete fires whenever any job reaches terminal; feature 17 is one consumer among many. Covered by the callback's own tests, not by feature-17 integration tests. |
| `runtime.RunRequest.IsService` skipping timeout enforcement | Tested at runtime layer (`go_runtime_test.go`). Cross-feature concern; re-pinning here duplicates the alarm. |
| 3xx redirect handling in the probe | Prober uses `client := &http.Client{Timeout: probeTimeout}` with default redirect policy (follow up to 10). A health endpoint returning 3xx is misconfigured by definition; Go's client behaviour is stdlib, not Helion code. |
| Oversize probe body beyond 4 KiB drain cap | Documented at `service_prober.go:87-91`. The cap is `io.CopyN(io.Discard, resp.Body, 4<<10)` — a Go stdlib guarantee. A test would time-bound a test httptest server that streams; the test would be timing-flaky while proving stdlib guarantees. |
| Submit→dispatch→probe→ReportServiceEvent→GET `/api/services` full chain | End-to-end; covered by forthcoming docker e2e. |
| Whitespace chars other than space (tab, CR, LF) | All go through the same `ContainsAny(s.HealthPath, " \t\r\n\x00")` branch. One representative covers the branch — peer byte classes would alarm on the same failure. |

Each evaluated against the audit framework bar — **would a real
regression actually trip this test?** and **is the alarm
qualitatively different from what's already in place?** Only
T1–T4 cleared both.

## Calibration note

Third feature-17 pass converged on branch-level gaps: four
`return` / emission lines whose removal wouldn't trip any
existing test. Pattern across the three passes:

- Pass 06 — integration-level alarm (the entire prober
  goroutine was untested).
- Pass 07 — mocked-contract alarm (the real audit logger was
  never tested with a real mock store).
- Pass 08 — branch-level alarms inside already-tested functions
  (the validator and response plumbing had tests, but specific
  branches slipped through).

Each pass sees finer granularity of gaps. This is the
"successive passes find qualitatively smaller things"
calibration from feature-12's audit history, confirmed again.
A fourth feature-17 pass would likely turn up only contrived
items (the tables above ran out of genuinely distinct branches
to test).

Recommendation: feature 18 (dashboard ML module) next. Six
prior audits (07, 09, 2026-04-18-01, 03, 05, 06) flagged
feature-18 as the consumer of every event whose producer side
was pinned — the dashboard audit closes the read side of those
contracts.

Pattern across features 11–17 now:

- 11: 4 passes → saturation
- 12: 4 passes → multiple production fixes
- 13: 2 passes → splitter divergence
- 14: 1 pass → event payload observer
- 15: 1 pass → API boundary
- 16: 3 passes → security + validator + audit-log
- 17: 3 passes → prober + audit/IPv6 + branch-level residuals
