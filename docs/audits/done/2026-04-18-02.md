## Test Gaps

### ~~T1 — `maxGPUs` validator boundary not pinned by any test~~ ✅ Fixed
**File:** `internal/api/handlers_jobs.go:214`

```go
if req.Resources != nil && req.Resources.GPUs > maxGPUs {
    return fmt.Sprintf("resources.gpus must not exceed %d", maxGPUs)
}
```

The cap is 16 (documented at line 57–62 with rationale:
commercial hosts ship 8 GPUs; 16 gives headroom for future
expansion). Existing `TestSubmitJob_WithResources_Returns201`
covers cpu_millicores / memory_bytes / slots but never sets
`gpus`. No test at any layer asserts:

- `gpus = 16` is accepted (inclusive upper bound).
- `gpus = 17` is rejected with 400.

A regression that flipped `>` to `>=` — a classic refactor
mistake when merging boundary-check conditions — would silently
reject every 16-GPU job. The only existing resource tests use
`slots: 2` / implicit 0 GPUs, so nothing in the test suite would
trip. This is the same class of silent regression audit
[2026-04-15-01](2026-04-15-01.md) and peers have pinned for
other caps (priority ≤ 100, workflow jobs ≤ 50,
cpu_millicores ≤ maxCPUCores * CPUPeriodUS). `maxGPUs` was the
odd one out.

**Fix:** two tests in
[`handlers_jobs_retry_test.go`](../../../internal/api/handlers_jobs_retry_test.go)
mirror the `TestSubmitJob_WithPriority_Returns201` /
`TestSubmitJob_PriorityTooHigh_Returns400` pair:

- `TestSubmitJob_GPUsAtMax_Returns201` — submits `{"gpus": 16}`,
  asserts 201.
- `TestSubmitJob_GPUsExceedsMax_Returns400` — submits
  `{"gpus": 17}`, asserts 400.

Both use the shared `newServer` / `do` helpers so they add one
line each to the package test surface; no new test infrastructure.

## Items considered and dismissed (not findings)

| Item | Why not |
|------|---------|
| Allocator double-release of the same `jobID` | `GPUAllocator.Release` at `gpu_alloc.go:81` iterates `busy` and deletes entries whose owner matches. A second Release with the same ID finds zero matches — the function is a no-op by construction. No guard is needed; no test is needed to pin "second call does nothing" because the iteration semantics make any other outcome impossible. Dismissed — not a gap. |
| Concurrent release-while-allocate on the same allocator | Both methods take the same `sync.Mutex`. The concurrent-distinct-allocations test already proves mutex ordering under load; adding a release-allocate interleave would exercise the same lock primitive with fancier scheduling. No new alarm. |
| Runtime Release on exec failure (`exec.ErrNotFound`, EACCES) | Go's `defer` runs on every function return path, including errors from `cmd.Start()` / `cmd.Run()`. `TestGoRuntime_GPURequest_Released_OnSuccess` + `..._OnFailure` in `gpu_runtime_test.go` cover both exit-code branches; the "fork itself failed" error path is the same defer, same Release call. Pinning it separately would test the Go runtime's defer semantics, not Helion's code. |
| Rust client: readFrame failure after writeFrame, mid-IPC | `RustClient.Run` uses a single `defer alloc.Release(jobID)` registered immediately after Claim succeeds (before `net.Dial`). Any error from dial / write / read / parse exits via the defer, same as the existing `TestRustClient_GPURequest_Released_OnSuccess`. A readFrame-fails test would exercise defer semantics through a longer failure path — same alarm, more setup. |
| `filterByGPU` + label selector — GPU-insufficient label-matched node | Already covered. `TestPickForJob_SelectorAndGPUCombined` in `scheduler_gpu_test.go` constructs three nodes: one label-match + zero GPUs, one no-label + 4 GPUs, one label-match + 4 GPUs. The request (selector + 2 GPUs) must pick only the third node. That exercises exactly the "label matches but GPU insufficient" path. |
| Heartbeat `TotalGpus` update path | `registry_heartbeat.go:54` updates capacity when any of `CpuMillicores > 0 \|\| TotalMemoryBytes > 0 \|\| MaxSlots > 0 \|\| TotalGpus > 0`. The atomic store covers the GPU update. An all-zero heartbeat (hypothetical: node reports no capacity at all) would skip the update — but real worker nodes always report `MaxSlots > 0` since that's configured at startup, never zero. The all-zero case is contrived. The persistence round-trip test proves the stored value survives restart; the heartbeat update is a thin atomic write with no branching logic worth separate test coverage. |
| `RoundRobinPolicy` per-candidate-set counter unbounded growth | Documented design choice: the `sync.Map` keyed by sorted-node-ID-list is deliberately uncapped because (a) candidate-set churn is bounded by the realistic number of distinct `(selector, GPU-count)` combinations in a cluster (≪ number of jobs), (b) per-entry overhead is ~100 bytes, (c) the alternative (LRU, time-based eviction) trades memory for contention on the hot path. If this becomes a real memory concern, it belongs in an explicit followup entry — not a test. Writing a test that creates 1000 synthetic candidate sets to document growth would pin a non-bug as behaviour. |
| Dashboard GPU-specific unschedulable reason | `classifyUnschedulable` doesn't distinguish "insufficient GPUs" from "no matching labels" — a GPU-only job with no selector and too-big request is classified as `no_matching_label` (empty selector fallback). That's an observability gap on the feature-18 side (richer reason for GPU dispatch failures); it belongs in the feature-18 audit, not feature-15. Audit [2026-04-18-01](2026-04-18-01.md)'s calibration note already flagged feature 18 as the next high-value audit target; this item becomes part of that pass. |
| Negative `gpus` input | `ResourceRequestAPI.GPUs` is `uint32` — JSON deserialisation rejects negatives at the wire before the validator runs. The existing wire-layer test surface covers JSON shape rejection; re-testing type conversion would exercise `encoding/json`, not Helion. |
| Real-GPU path (`runNvidiaSmi` / `runNvidiaSmiCount`) | Explicitly deferred to the `tests/gpu/` harness under `//go:build gpu`, run locally on hardware. Out of scope for the CI-runnable audit loop per feature-15's explicit testing split. |

Each evaluated against the two honest questions from
[2026-04-15-02](done/2026-04-15-02.md)'s framework — **would a
real regression actually trip this test before production?**
and **is the alarm qualitatively different from what's already
in place?** Only T1 cleared both bars.

## Calibration note

Feature 15 has the largest test surface of any ML-pipeline
slice audited so far (19 pure-Go + 2 hardware tests, 8 distinct
test files across 4 packages). The first-pass coverage is
strong because the feature's production bug surface was
literally walked through by its own second-pass audit
(referenced in the spec: `CUDA_VISIBLE_DEVICES` override safety,
CPU-job-on-GPU-node hide, persistence roundtrip). What this
audit found is an HTTP-layer boundary gap — not something the
feature's own audit would have uncovered because it focused on
runtime / scheduler / persistence.

Pattern observation across features 11–15:
- Feature 11 (4 passes): saturation after cross-backend contract
  lock.
- Feature 12 (4 passes): real production-code fix on pass 3
  (`attestOutputs` declared-name cross-check).
- Feature 13 (2 passes): real production-code fix on pass 2
  (last-dot splitter divergence).
- Feature 14 (1 pass): event-payload field observer gap.
- Feature 15 (1 pass): API boundary-validator gap.

The gap *class* shifts down the severity ladder as the features
get smaller and more focused. Features that cross package
boundaries with sibling helpers (feature 13) harbour divergence
bugs; features with multi-field event payloads (feature 14)
harbour observer gaps; features with validators (feature 15)
harbour boundary gaps. Each audit pass plays to its feature's
shape. Don't expect the same finding class to repeat — adjust
the search.

Recommendation: feature 18 (dashboard ML module) next. Two
prior audits (07, 09) have flagged feature-18 observability
concerns: the ml.resolve_failed / job.unschedulable event
payloads the dashboard reads, and the GPU-specific reason
classification that's missing from `classifyUnschedulable`. A
focused feature-18 pass would close those cross-cutting
threads.
