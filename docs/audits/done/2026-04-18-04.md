## Findings

### ~~L1 — `ValidateMetrics` over-broad finite rejection via 1e308 threshold~~ ✅ Fixed
**File:** `internal/registry/validate.go:172-177`

The pre-fix code was:

```go
if v != v { // NaN check
    return fmt.Errorf("metric %q is NaN", k)
}
if v > 1e308 || v < -1e308 {
    return fmt.Errorf("metric %q is infinite", k)
}
```

The comment above it says:

> NaN and ±Inf can't round-trip through the JSON encoder
> (encoding/json rejects them).

But `v > 1e308` is stricter than ±Inf detection. `math.MaxFloat64`
is approximately `1.7976931348623157e308` — a perfectly finite
value that `encoding/json` serialises cleanly. The old check
rejected it with error `"metric %q is infinite"`, a misleading
error message for a finite value.

Two small problems compound:

1. **Comment/code mismatch** — future maintainer reads the
   comment, understands "only NaN + ±Inf are rejected", but the
   code rejects more. Any follow-up tightening / loosening would
   start from the wrong baseline.
2. **Misleading error text** — a value of `1.5e308` returns
   `"metric foo is infinite"` when it isn't. A frustrated user
   debugging this has no way to guess the real threshold.

In practice no real ML workload submits metrics above 1e308, so
this is a purely theoretical impact on submitted data — but the
audit loop has consistently flagged comment/code divergences as
real because they corrupt future decision-making.

**Fix:** switch to the Go-idiomatic `math.IsNaN` / `math.IsInf`:

```go
if math.IsNaN(v) {
    return fmt.Errorf("metric %q is NaN", k)
}
if math.IsInf(v, 0) {
    return fmt.Errorf("metric %q is infinite", k)
}
```

The existing NaN / +Inf / -Inf test cases continue to pass
because `math.IsInf(v, 0)` catches both signs. The new finite
extremes (MaxFloat64, -MaxFloat64, 1.5e308) now validate
cleanly, matching the comment's promise.

### ~~T1 — `MaxFloat64` / finite-extreme metric values unpinned~~ ✅ Fixed
**File:** `internal/registry/validate_test.go`

The existing `TestValidateMetrics` covers the happy path
(0.95, 0.87), NaN, ±Inf, and empty-key — but no assertion pins
the behaviour for finite extremes. Without a test for
`math.MaxFloat64`, a regression reintroducing the 1e308 threshold
check (the bug L1 just fixed) would slip past every existing
metric test. This is the "regression that would not trip any
test" class — identical to the audit-framework's primary alarm
criterion.

**Fix:**
[`validate_test.go:TestValidateMetrics_FiniteExtremesAccepted`](../../../internal/registry/validate_test.go)
submits three values that would trip a 1e308-style threshold
but validate cleanly under `math.IsInf`:

- `math.MaxFloat64` (≈ 1.7976e308)
- `-math.MaxFloat64`
- `1.5e308` (explicitly in the regression window between 1e308
  and MaxFloat64)

All three must pass `ValidateMetrics` with no error.

## Items considered and dismissed (not findings)

| Item | Why not |
|------|---------|
| `ValidateName` allows `.` and `..` and leading/trailing `-` or `.` | Name is a BadgerDB key segment, not a filesystem path — no path traversal risk since BadgerDB's LSM doesn't normalise. The URL router rejects `//` collapsing so `/api/datasets/./v1` would 404 at the routing layer before hitting the handler. k8s-style DNS labels allow leading alnum only (stricter); this registry is deliberately laxer because the name space is Helion-internal and never reaches a DNS resolver. A test would pin a laxity choice, not alarm for a regression. |
| `ValidateURI` allows `file://` URIs without an absolute path check | The scheme allowlist (`file://`, `s3://`) matches `validateArtifactBindings` at the API boundary. Both accept `file://relative/path` — a relative file URI is malformed per RFC 8089 but downstream consumers (Stager, artifact store) fail loud on the missing path resolution. Adding a registry-side absolute-path check would duplicate what the ArtifactStore's `Open` already alarms for. |
| Dataset `SizeBytes` accepts negative values | The field is `int64` — JSON accepts negative numbers into it. Downstream consumers (dashboard storage-aggregation query) treat sizes as non-negative; a negative value would skew aggregates. But: the registered size comes from `artifact.Stat` (an ArtifactStore call that returns `Info{Size int64}` with non-negative invariant by construction), so a legitimate caller never passes a negative. A malicious caller with JWT authority to register can already register arbitrary metadata (this is the "registrar is trusted to stamp correct pointers" trust model documented in the feature doc). Adding a non-negative assertion would alarm for a violation of a trust-model axiom, not a regression. |
| Dataset `SHA256` format not validated | Same reasoning as SizeBytes — the digest comes from `artifact.Stat`, registered faithfully, consumed by downstream code that hash-compares the fetched bytes. A malformed SHA would fail verification downstream, not silently corrupt state. Validating the SHA shape in the registry would duplicate the artifact store's `GetAndVerify` alarm. |
| Audit log for `model.registered` doesn't include `source_dataset` but the event bus payload does | The audit log at `handlers_registry.go:280` records `name/version/uri/source_job_id` only. The event bus payload at `ModelRegistered` adds `source_dataset.{name, version}`. Inconsistent, but the audit log's taxonomy is documented in the feature spec's audit event taxonomy table as including `source_job_id` only — the event payload goes further for the analytics pipeline. Deliberate split, not a bug. If the audit-query UI ever needs to filter by source dataset, that's a spec-level change not a test gap. |
| Second DELETE returns 404 not 204 — idempotence semantics | Already dismissed in audit 03 with the same reasoning and still holds: DELETE of a missing resource exercises the same `ErrNotFound` translation as GET of a missing resource, which is thoroughly covered. |
| BadgerStore prefix scan collision between `models/foo` and `models/foo-bar` names | Verified by byte-by-byte comparison: `models/foo/` (prefix) vs `models/foo-bar/v1` (key). Byte 10 differs (`/` vs `-`), so no collision. BadgerDB's prefix iterator is byte-literal, not normalised. No test needed to pin a behaviour the data structure guarantees. |
| Symmetric partial-lineage rejection — `SourceDataset{Version: "v2"}` with empty Name | The existing `TestValidateModel_PartialLineage_Rejected` covers the name-set-version-empty half. The validator's branch condition (`Name != "" \|\| Version != ""`) activates for either half; the inner `ValidateName` / `ValidateVersion` calls then reject the empty half. Both halves hit the same code path. A symmetric test would exercise the same code with mirrored input — coverage without new alarm. |
| Metric keys containing NUL | Rejected at `validate.go:166-168`; not explicitly in the metrics test. Symmetric to the tag NUL-key rejection that IS tested. Low value. |
| Page size of exactly 0 in `parsePageSize` | `parsePageSize` silently falls back to default 20 when Atoi succeeds but value ≤ 0. The existing list tests exercise defaults (no query params → happy path). Explicit `?size=0` would exercise the strconv branch under a specific input — exercising fallback, not reporting behaviour. |
| `ListBySourceJob` linear scan for models | Documented explicitly at `badger.go:146-151`: "Linear scan; acceptable at MVP scale because the caller is behind the registry rate limiter and model counts in a minimal-ML deployment are O(hundreds)." Scale concern; not a regression target. |
| `LatestModel` with a name prefix of an existing registered name — e.g. `LatestModel("a")` when `"ab"` exists | Verified above — no collision, BadgerDB prefix semantics byte-literal. |

Each evaluated against the two honest questions — **would a
real regression actually trip this test before production?**
and **is the alarm qualitatively different from what's already
in place?** Only L1 / T1 cleared both.

## Calibration note

Second feature-16 pass found one real code bug that the first
pass missed (and whose existence the first pass had implicitly
mis-dismissed by reasoning about what the comment claimed the
code did, rather than what the code actually did). The pattern
matches the feature-12 trajectory: the second or third pass
tends to find smaller, more specific issues by going below the
pattern-search layer into line-by-line reading.

Audit 03 recommended feature 18 (dashboard) next. That still
holds — feature 16 is now saturated for practical purposes.
Further passes on feature 16 are unlikely to find more than
hygiene items (symmetric test cases, laxity choices) that the
audit framework consistently dismisses.

Pattern across features 11–16, post-second-pass-where-applicable:

- 11: 4 passes → cross-backend contract + coverage saturation
- 12: 4 passes → 1 production fix + operational gaps
- 13: 2 passes → 1 production fix (splitter divergence)
- 14: 1 pass → event-payload observer
- 15: 1 pass → API boundary
- 16: 2 passes → 3 security/observability gaps (pass 1) + 1 validator bug (pass 2)

Features with validator helpers reward a second pass because
validators get written once, tested for obvious cases, and
then quietly drift as comments are updated without the code
following. Line-by-line re-reading is the only reliable way to
catch that drift.
