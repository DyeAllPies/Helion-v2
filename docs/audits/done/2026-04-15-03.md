## Test Gaps

### ~~T1 ŌĆö `ReportResult` RPC didn't have a test proving attestation actually runs in the handler~~ Ō£ģ Fixed
**File:** `internal/grpcserver/handlers.go:ReportResult` ŌĆö the
single line `opts.ResolvedOutputs = s.attestOutputs(...)`

`attestOutputs` itself is well-tested in
[`internal/grpcserver/security_test.go`](../../../internal/grpcserver/security_test.go)
(prefix mismatch, count cap, shape rules). What no test covered was
**whether the ReportResult handler actually pipes the reported
outputs through it**. A regression that swapped
`s.attestOutputs(...)` for `result.Outputs` directly ŌĆö say, a
future "refactor to skip an extra copy for performance" ŌĆö would
slip past every unit test because the validator itself would still
work; the bypass would only surface in production when a
compromised node successfully injected foreign URIs into another
job's record.

**Fix:**
[`report_result_test.go:TestReportResult_AttestsOutputs_DropsForgedEntries`](../../../internal/grpcserver/report_result_test.go)
fires a ReportResult over the real gRPC socket with two outputs:
one legitimate (URI under `jobs/<this-job>/ŌĆ”`) + one forged (URI
under `jobs/<other-job>/ŌĆ”`). The `mockJobStore.Transition`
captures the call's `TransitionOptions.ResolvedOutputs`; the test
asserts exactly the legitimate one made it through. If anyone ever
stops calling `attestOutputs` at the handler level, the forged URI
will reach the store and the test fails loudly.

Required one small helper extension ŌĆö `mockJobStore` grew a
`lastTransitionOpts` field so tests can observe the slice the
handler passes down. Same mock now mirrors ResolvedOutputs onto
the stored Job so the assertion can also be spelled against
`js.Get(jobID).ResolvedOutputs` for readers who prefer that shape.

### ~~T2 ŌĆö Stager upload path had no live-MinIO integration test~~ Ō£ģ Fixed
**File:** `internal/staging/staging.go:Stager.Finalize`

The Stager's unit tests (25 of them in
[`internal/staging/staging_test.go`](../../../internal/staging/staging_test.go))
all use `LocalStore`. The live-MinIO integration tests added in
audits 2026-04-15-01/02 exercise the Store directly, not through
the Stager. What was missing is the path that matters in
production: **a real Stager against a real S3Store uploading a
real output file and producing a URI the coordinator's attestation
will accept**.

This gap is narrow but load-bearing. The coordinator's
`uriBelongsToJob` check requires `s3://<bucket>/jobs/<job_id>/<local_path>`;
if the Stager's Finalize ever built a URI with a slightly
different shape ŌĆö extra slash, missing prefix, bucket name drift ŌĆö
every output would be silently dropped on attestation, jobs would
still transition to Completed, but the Pipelines DAG view would
show no `models_produced` and the iris demo would never populate
the registry. Hard to diagnose without a live test.

**Fix:**
[`tests/integration/artifacts_live_s3_test.go:TestLiveS3Stager_UploadsOutputOnFinalize`](../../../tests/integration/artifacts_live_s3_test.go)
runs the full `Prepare ŌåÆ write file in workdir ŌåÆ Finalize` loop
against a real S3Store pointed at live MinIO (gated on
`MINIO_TEST_ENDPOINT`, picked up by `scripts/run-e2e.sh`'s
live-MinIO block). Asserts three invariants:

  1. Finalize returns a URI ending `/jobs/<job_id>/<local_path>` ŌĆö
     the exact shape attestation's `uriBelongsToJob` checks for.
  2. The URI resolves ŌĆö bytes fetched from MinIO match what the
     job wrote into the workdir.
  3. A Finalize call for a failed run uploads **nothing**; the
     would-be URI returns `ErrNotFound` on Stat, proving the
     "upload only on success" contract the feature 12 spec
     promises.

## Items considered and dismissed (not findings)

| Item | Why not |
|------|---------|
| Rust runtime `working_dir` parity integration test | The feature doc (line 184) notes parity was closed by commit `506680d` and that "input/output bytes flow through the Stager at the nodeserver level, not through the runtime, so no further Rust-side work is required." Writing a Rust-runtime integration test would duplicate the Go-runtime tests; the cross-runtime concern is that `cmd.current_dir()` is honored, which is a single line the Rust tests already exercise. |
| Node-startup sweep integration test | `internal/staging/sweep_test.go` covers the sweep behaviour thoroughly; verifying that `cmd/helion-node/main.go` actually calls it at startup is wiring-check territory better caught by a full-cluster e2e run (queued for the batch at the end of the 11ŌĆō19 sweep). |
| Parallel jobs on same node not contaminating each other's workdirs | Structural ŌĆö each job gets a separate `Mkdir` under the work root, `safeJoin` re-validates every path against that root, and the existing `TestPrepare_ConcurrentDistinctJobs` exercises the common path. No production failure mode clears the "would this test find a real regression?" bar. |
| Dashboard submit-with-inputs UI test | The `/jobs` submit form in the dashboard doesn't expose the Inputs/Outputs fields (it's a simple command submitter). The full ML submit flow goes through `POST /workflows` YAML, which the iris demo (feature 19) exercises end-to-end. No meaningful UI surface to test. |
| Environment-variable shadowing round-trip | `TestEnvPrecedence_StagerOverridesUserEnv` already proves the stager-wins rule at the merge layer. An integration test that additionally starts a real runtime to echo `$HELION_INPUT_X` adds process-spawn overhead without a different alarm. |
| `HELION_KEEP_WORKDIR=1` debug-flag behaviour | `TestPrepare_KeepFlag_PreservesWorkdir` covers it; nothing upstream changes behaviour based on this flag. |
| Staging audit event emission (`staging.prepared` / `staging.uploaded` / `staging.failed`) | Feature doc explicitly marks these as **taxonomy designed, emission deferred**. Testing their emission would fail by design ŌĆö they're not emitted yet. |
