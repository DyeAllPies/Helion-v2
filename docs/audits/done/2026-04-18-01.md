## Test Gaps

### ~~T1 — `job.unschedulable` event `reason` field wiring not asserted by any observer test~~ ✅ Fixed
**File:** `internal/cluster/dispatch.go` — the three call sites of
`d.maybeEmitUnschedulable(job, <reason>)` at lines 261, 272 (and
the eligibility-gated 254 path via `classifyUnschedulable`).

`events.JobUnschedulable` carries a `reason` field that the
feature-18 Pipelines dashboard uses to distinguish three
operationally-distinct stall causes:

- `no_healthy_node` — wait / retry (transient)
- `no_matching_label` — operator must add a matching node or
  change the selector
- `all_matching_unhealthy` — restart the stale matching nodes

`classifyUnschedulable`'s internal unit test pins the
classification, and the three integration tests
(`SelectorNoMatch_EmitsUnschedulable`,
`UnschedulableEvent_DebouncedPerJob`,
`UnschedulableEvent_RecoversAfterLabelAdded`) prove the event
fires — but none of them inspect `evt.Data["reason"]`. The
wiring between the helper and the bus payload is therefore
unpinned: a regression that always passed the wrong constant,
or collapsed the reason into a single value, or dropped the
key, would leave every test green while silently breaking the
dashboard's triage view.

This is the same class of silent regression audit
[2026-04-15-07](2026-04-15-07.md) fixed for the
`ml.resolve_failed` event — "the event fires, the payload
lacks a load-bearing field, dashboard silently misclassifies."

**Fix:**
[`dispatch_unschedulable_test.go:TestDispatch_UnschedulableEvent_ReasonField_Correct`](../../../internal/cluster/dispatch_unschedulable_test.go)
runs two sub-tests against the real `DispatchLoop`:

- `no_matching_label` — Registry has a healthy CPU node, job
  selector requests `gpu=a100`. Asserts
  `evt.Data["reason"] == events.UnschedulableReasonNoMatchingLabel`.
- `no_healthy_node` — empty Registry, job has no selector.
  Asserts `evt.Data["reason"] == events.UnschedulableReasonNoHealthyNode`.

Third classification (`all_matching_unhealthy`) is covered by
the `classifyUnschedulable` unit test in
[`dispatch_unschedulable_reason_test.go`](../../../internal/cluster/dispatch_unschedulable_reason_test.go)
and shares the identical emit path through the same
`maybeEmitUnschedulable` helper — adding a third integration
test would duplicate the alarm without exercising new code.

## Items considered and dismissed (not findings)

| Item | Why not |
|------|---------|
| Boundary tests for label size caps (key at 63 vs 64, value at 253 vs 254, map at 32 vs 33 entries) | The existing `TestRegistry_DropsInvalidLabels` covers oversize values via `strings.Repeat("v", 300)`, and the caps are set to k8s-compatible defaults comfortably above real-world need (operators typically set ≤5 labels per node). A regression that flipped `>` to `>=` in `validLabelKey` / `validLabelValue` would be caught by any existing size-reject test because `300 > maxNodeLabelValLen` is still true under `>=`. An exact-boundary test would only alarm for a flip from `>` to `<`, which is an implausible refactor. Dismissed as contrived. |
| Concurrent Register + PickForSelector race on label replacement | The node label map is stored behind `atomic.Pointer[map[string]string]` — by construction, each scheduler snapshot gets either the pre- or post-replace map atomically. A "torn read" cannot happen. The only operator-visible behaviour is "scheduler sees old labels once, then new labels on the next tick", which is a valid linearisation of concurrent events. No test can distinguish a valid ordering from an invalid one when both produce the same observable result. Dismissed. |
| Debounce cleanup on label change (node relabels to match a stuck selector) | Already covered by `TestDispatchLoop_UnschedulableDebounceClearsOnPick` (internal test asserting the `delete(d.unschedulableLastEmit, job.ID)` at dispatch.go:279 fires on successful pick) + `TestDispatch_UnschedulableEvent_RecoversAfterLabelAdded` (integration test that drives the end-to-end recovery). The combination of unit + integration proves the cleanup both fires *and* is reached via the real loop path. |
| `on_failure` + URI-only inputs (no `from:`) validation | Already implicitly covered — `validateFromReferences` at `dag.go:138` checks `if in.From == "" { continue }` before the condition gate runs. A URI-only input exits the loop body on line 138; no code path reaches the rejection. `TestValidateDAG_OnFailure_WithoutFromRef_Allowed` (empty inputs) proves the positive case; adding a URI-only variant would exercise the same `continue` branch with more boilerplate. Dismissed. |
| Nil vs. empty `Node.Labels` map semantics | `sanitiseNodeLabels` returns `nil` when the input is empty or all entries are dropped (documented at line 51). The snapshot reader unconditionally unwraps the atomic pointer and range-iterates, which is identical for nil and empty. `nodeMatchesSelector` returns false when `n.Labels == nil` and the selector is non-empty — already covered by `TestPickForSelector_NilLabelsOnNodeWithSelector`. A persistence round-trip of an all-dropped labels node is the only un-tested case, and that's just re-asserting forward-compat already covered by the feature's `persistence_labels_test.go`. |
| HELION_LABEL_* env with empty value (`HELION_LABEL_FOO=`) | The node-side `gatherNodeLabels` does not check for empty values — it would forward `foo=""`. The coordinator-side `sanitiseNodeLabels` keeps empty values (value length 0 is within the ≤253 cap; value charset check allows empty). So an operator-set empty label reaches the scheduler. This is intentional — an empty value is still a valid selector target (`gpu=""` matches "no GPU"). The existing `TestGatherNodeLabels_EnvOverridesAutoDetection` proves env overrides work; exhaustive variants of the value shape are YAGNI. |
| `PickForJob` returning `ErrNoNodeMatchesSelector` when the Policy returns nil for a non-empty slice | This is a defensive guard against a buggy Policy implementation. All three shipping policies (RoundRobin, LeastLoaded, ResourceAware) have unit tests proving they return a non-nil pick for a non-empty slice. Adding a "buggy policy returns nil" test would exercise a hand-written bad stub, not production code. |
| Unicode in label keys / values | The validators check for bytes 0x00, C0 (<0x20), and 0x7f — explicitly byte-level to dodge normalisation ambiguity. Multi-byte UTF-8 chars pass through as-is, and selector matching is byte-level `==`, so "é" on a node matches "é" on a selector iff the byte sequences are identical. This is by design (k8s-lite) and covered by the spec. A test that verified this would pin a behaviour the code didn't claim to formalise. |

Each evaluated against the audit-02 framework — **would a real
regression actually trip this test before production?** and
**is the alarm qualitatively different from what's already in
place?** Only T1 cleared both bars.

## Calibration note

Feature 14 is structurally smaller than feature 12 but larger
than feature 13 — it touches node agent, coordinator
sanitiser, scheduler, dispatch loop, DAG validator, and
persistence. This first pass found one gap that matches the
feature 07 pattern exactly: a load-bearing event-payload field
with no observer test. The bug class is consistent (event-emit
call site + multi-field payload + dashboard consumer =
silent-regression risk on any one field) so the fix pattern is
consistent too.

The recurring pattern is now clear enough to generalise:
**whenever a coordinator emits an event with three or more
operator-visible fields, pin each load-bearing field by an
observer test at the emit site.** Applies to `TopicJobUnschedulable`,
`TopicMLResolveFailed`, `TopicJobFailed` (not yet audited). The
feature 14 + 13 pattern suggests feature 18 (dashboard) —
which consumes both of these events — should be the next audit
target: what other load-bearing fields does it read without an
observer test pinning the write?

Recommendation: feature 15 (GPU first-class resource) next
unless review priority favours feature 18 (dashboard)
specifically because of the cross-cutting observer-test
pattern.
