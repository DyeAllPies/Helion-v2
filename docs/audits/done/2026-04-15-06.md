## Test Gaps

### ~~T1 ‚Äî `NewStager` empty-`workRoot` fallback to `os.TempDir()/helion-jobs` had no test~~ ‚úÖ Fixed
**File:** `internal/staging/staging.go:68-70`

The Stager constructor documents a fallback: if the caller passes
an empty `workRoot` (the default "`HELION_WORK_ROOT` unset" config
in production), the Stager synthesises one under
`filepath.Join(os.TempDir(), "helion-jobs")`. Every unit test in
the package passes an explicit `workRoot` via `t.TempDir()`, so
the fallback path is uncovered ‚Äî a regression that renamed the
directory, changed the base dir, or removed the fallback
altogether would leave every test passing while production nodes
either picked a surprising location or crashed at first Prepare.

Low value but real: production default gets a test.

**Fix:**
[`staging_test.go:TestNewStager_EmptyWorkRoot_FallsBackToTempDir`](../../../internal/staging/staging_test.go)
passes `""` as `workRoot`, runs Prepare, and asserts the resulting
`p.WorkingDir` resolves under `os.TempDir()/helion-jobs`.
Case-insensitive prefix match on Windows where TempDir casing
drifts across environments.

## Items considered and dismissed (not findings)

| Item | Why not |
|------|---------|
| `.helion-stage-*.tmp` cleanup on verified-download error | Covered implicitly by Prepare's workdir-level rollback ‚Äî `TestPrepare_VerifiedDownload_TamperDetected` already asserts the entire subtree is gone, which subsumes any leaked tempfile inside it. Adding a tempfile-specific assertion would be redundant. |
| Node-ID cross-check on `ReportResult` mismatch | Already covered by `internal/grpcserver/node_id_mismatch_test.go:TestReportResult_NodeIDMismatch_Rejects` + `TestReportResult_NodeIDMatches_Accepts`. Two clean tests from a prior audit. |
| `HELION_OUTPUT_*` env values point at staged output paths | Covered by `TestPrepare_InputsStaged_EnvExported` (asserts the env value is absolute; the Stager's implementation guarantees it lives under the workdir). Redundant to add a second alarm. |
| `workRoot` already exists with wrong mode bits (0o755 instead of 0o700) | `os.MkdirAll` is a no-op on existing dirs and doesn't tighten the mode. Real concern ‚Äî but this is an operator-misconfiguration scenario, not a code-regression one. The `0o700` on fresh Prepare is already covered by `TestLocalStore_Permissions` (for the Store root) and the Stager's own MkdirAll at line 212. An existing-dir permission widener would be a deployment bug, not something a test can alarm for. |
| Concurrent `Prepare` + `Finalize` on the same job | The dispatch loop is single-threaded per job (`dispatchPending` iterates jobs one at a time and each job only has one lifecycle). A second Prepare would collide on the existing workdir ‚Äî which would fail loud. No test needed for a scenario the dispatcher's design rules out. |
| Stager with a non-existent explicit `workRoot` that can't be created | Covered indirectly ‚Äî `MkdirAll` would return an error, which Prepare wraps and returns. Any of the sibling "Prepare rolls back on failure" tests would catch a regression in the error path. |
| `Finalize` with zero Outputs declared (batch job case) | The happy path of most existing tests (TestDispatch_Success etc.) uses no-Outputs jobs. Zero-outputs Finalize is the common batch-job path ‚Äî covered by the broad surface. |

Each evaluated against the two honest questions from the
2026-04-15-02 framework: **would a real regression actually trip
this test before production?** and **is the alarm qualitatively
different from what's already in place?** Only one cleared both
bars; six didn't.
