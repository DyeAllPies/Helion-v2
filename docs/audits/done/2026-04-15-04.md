## Medium

### ~~M1 — `MaxOutputUploadBytes` rejection path had no test~~ ✅ Fixed
**File:** `internal/staging/staging.go:409-411` — the
pre-flight `Lstat`-size check in `Stager.upload`.

Feature 12's threat matrix explicitly lists "Oversized output
exhausts store" as a threat, mitigated by a
`MaxOutputUploadBytes` (2 GiB) pre-flight Lstat. The check exists
in code; no test ever hit the rejection branch. Meanwhile the
parallel input-side cap (`MaxInputDownloadBytes`) does have
[`TestPrepare_InputSizeCapEnforced`](../../../internal/staging/staging_test.go).
Asymmetric coverage on a symmetric threat — if a refactor deleted
the output check (e.g. "the API validator already caps URIs, why
do we need this?"), every test would still pass and a runaway
training job could fill the artifact store.

**Fix:**
[`staging_test.go:TestFinalize_OversizeOutput_Rejected`](../../../internal/staging/staging_test.go)
lowers the cap globally to 1 KiB (the package doc explicitly
allows this — the var is documented as test-tweakable), writes a
2 KiB output file, and asserts `Finalize` returns an error whose
message contains "cap". Also walks the LocalStore root and
confirms **nothing was uploaded** — an earlier faulty fix that
validated then uploaded anyway would fail this second assertion.

## Low

### ~~L1 — Stager-less node refusal of artifact bindings had no test~~ ✅ Fixed
**File:** `internal/nodeserver/server.go:98-101` — the
`if s.stager == nil { …refuse… }` branch at the top of
`Dispatch`.

A node started without `HELION_ARTIFACTS_BACKEND` runs with
`stager == nil`. The Dispatch handler refuses any job that
declares Inputs / Outputs / WorkingDir rather than running the
command with bindings silently ignored (the feature 12 spec's
fail-closed rule). Without this guard, a misconfigured node
would accept a workflow job, run the command, upload nothing —
the coordinator's resolver (feature 13) would then time out
waiting for outputs that never arrived, the iris demo would
hang, and the operator would have no obvious "your node isn't
configured for ML" signal. The guard is load-bearing in
production; a regression on it produces exactly the kind of
silently-wrong behaviour that's worst to debug.

**Fix:**
[`server_test.go:TestDispatch_StagerLess_RefusesArtifactBindings`](../../../internal/nodeserver/server_test.go)
— table-driven over three declaration shapes (Inputs only,
Outputs only, WorkingDir only), each asserts `Accepted=false`
and the refusal message contains "stager". Uses the existing
`newServer(rt)` helper which already passes `stager=nil`, so the
wiring being tested is exactly what a misconfigured production
node would see.

## Test Gaps

### ~~T1 — `Prepared.Cleanup` double-call had no test~~ ✅ Fixed
**File:** `internal/staging/staging.go:96-101` — the
`if p != nil && p.cleanup != nil { p.cleanup(); p.cleanup = nil }`
double-call guard.

Defer-based cleanup + manual test-level Cleanup routinely call
this twice in practice (search for `p.Cleanup()` in the existing
test file — several spots already exhibit the pattern). The
guard is correct; nothing enforces that it stays correct. A
regression that dropped the nil guard or the `p.cleanup = nil`
reset would produce a panic on the second call — often in a
defer, which would hide the real test failure behind a panic
trace.

**Fix:**
[`staging_test.go:TestPrepared_Cleanup_Idempotent`](../../../internal/staging/staging_test.go)
runs Prepare, calls `Cleanup()` explicitly, verifies the workdir
is gone, then calls `Cleanup()` a second time under a subtest
that recovers from panic. The subtest passes only when the
second call is a silent no-op — any panic fails it cleanly.

## Items considered and dismissed (not findings)

| Item | Why not |
|------|---------|
| End-to-end workflow artifact chain (two-job DAG with `from: A.OUT` consuming A's output via live MinIO Stager) | Straddles feature 12 (Stager) and feature 13 (workflow resolver); belongs to feature 13's audit pass, where the resolver's contract is the primary subject. Testing it here would duplicate coverage. |
| Concurrent Finalize (two jobs uploading at once) | LocalStore's concurrent-Put-distinct-keys case is covered by feature-11's `TestConcurrentPuts_DistinctKeys`. Stager-level concurrency adds no new contention: each job uses its own `jobs/<id>/` prefix, so backend contention is what's already tested. |
| Streaming-download path against live MinIO in the Stager (input side) | The feature-11 live-MinIO test covers the Store's streaming Get; the Stager just wraps `GetAndVerifyTo` on top. Adding a Stager-layer live test would duplicate feature-11's `TestLiveS3GetAndVerifyToStream` without exercising new code. |
| Store-error mid-upload (transient S3 failure partway through Put) | The per-backend tests for `TestS3_PutError_Propagated` cover the error-propagation shape; the Stager just forwards the error. No new alarm. |
| Workflow artifact validation differs from plain-submit validation | `TestWorkflowAPI_Submit_*` already covers this; the validator is shared. |
| `safeJoin` fuzz test | The existing `TestSafeJoin` table covers the adversarial inputs explicitly (traversal, absolute, encoded, drive letters). Fuzzing would find random invalid inputs that also get rejected — noisy without a real alarm. |
| Prepare on a store that returns a ReadCloser erroring mid-read | `TestPrepare_VerifiedDownload_TamperDetected` and `TestPrepare_WrongDigest_ContentRejected` exercise adjacent failure shapes; a new case for "reader errors after the first few bytes" is marginal. |

Each considered against the same two questions as
[`2026-04-15-02`](2026-04-15-02.md): **would a real regression
actually trip this?** and **is the alarm qualitatively different
from what's already in place?** Three passed, seven didn't.

feature 12 coverage is saturated at the same level feature 11

auditing on this feature will find contrived gaps, not real ones.
