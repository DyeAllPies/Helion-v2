transition classes (unknown‚Üíready, ready‚Üíunhealthy) since that's

## Test Gaps

### ~~T1 ‚Äî Prober state machine entirely untested (edge-triggered invariant)~~ ‚úÖ Fixed
**File:** `internal/nodeserver/service_prober.go:96-113` ‚Äî the
probe loop's `if !haveLastReady || ready != lastReady` guard.

The prober was shipping without any test file. The invariant "a
happy service emits exactly ONE `service.ready` event across
its entire lifetime" rests on a single conditional; dropping
the `ready != lastReady` half would spam the audit log with one
emit per 5s tick per ready service. Production consequences:

- Audit log bloat (cardinality ‚àù uptime √ó service count).
- Downstream dashboard noise ‚Äî the Pipelines view would show
  readiness flicker even when nothing changed.
- Bus congestion if multiple ready services ran on the same node.

**Fix:** new
[`service_prober_test.go`](../../../internal/nodeserver/service_prober_test.go)
wires a real coordinator gRPC server + real `grpcclient` from
the node side + `httptest.NewServer` as the health target.
Drops `probeInterval` to 25 ms so 5+ ticks fit in a 175 ms
observation window.

Two sub-tests:

- `TestProbeService_HappyPath_EmitsSingleReadyEvent` ‚Äî health
  endpoint always returns 200, observes ‚â•5 ticks, asserts
  **exactly 1** service event fired (not `<=1` ‚Äî so a regression
  that skipped the initial emit also fails). Asserts registry
  shows `Ready=true`.
- `TestProbeService_ReadyToUnhealthyTransition_EmitsOnFlip` ‚Äî
  health endpoint starts 200, flips to 500 after 80 ms, observes
  another 200 ms. Asserts **exactly 2** events across the two
  states and registry `Ready=false` at the end.

### ~~T2 ‚Äî `LogServiceEvent` audit emission unpinned on the happy path~~ ‚úÖ Fixed
**File:** `internal/grpcserver/handlers.go:545-551`

The handler emits an audit entry via `s.audit.LogServiceEvent`
on every accepted ReportServiceEvent RPC. The mock audit
logger's `serviceEvents` counter existed in `testhelpers_test.go`
but no test asserted the counter incremented. This is the same
silent-regression class surfaced in feature-16 audit 05 for the
registry handlers' audit calls.

**Fix:**
[`TestReportServiceEvent_UpsertsRegistry`](../../../internal/grpcserver/report_service_event_test.go)
now asserts `al.serviceEvents == 1` after the RPC. The assertion
is `==1` (not `>=1`) so a regression adding a second emit ‚Äî
e.g. a refactor that called the helper twice per event ‚Äî also
fails.

### ~~T3 ‚Äî `LogServiceEvent` coverage via the integration path~~ ‚úÖ Fixed
**File:** same as T1.

The new prober tests also exercise the
`grpcserver ‚Üí LogServiceEvent ‚Üí mock audit logger` chain
end-to-end. Each asserted `serviceEvents` count (1 for the
happy path, 2 for the flip) is both the state-machine alarm and
the audit-emission alarm ‚Äî a regression in either place trips
the same assertion. The doubly-covered nature is intentional:
a failure message that says "expected 1, got 2" immediately
points at the edge-trigger regression, while "expected 1, got
0" points at the audit emission regression. Two distinct
failure signals from one assertion.

## Items considered and dismissed (not findings)

| Item | Why not |
|------|---------|
| Path-traversal / query-string injection in `HealthPath` | The prober binds to `127.0.0.1` and targets only the user's own service running in the same pod/node. A path like `/../admin` reaches the user's own service ‚Äî they control both ends. An attacker with submit authority can already spawn any command and probe any loopback path directly. The validator's current checks (absolute path, ‚â§256 bytes, no whitespace/NUL) are sufficient for syntactic safety; semantic path restrictions would over-constrain legitimate health endpoints like `/healthz?ready=deep`. |
| Grace period negative value | `HealthInitialMs` is `uint32` ‚Äî JSON decoding rejects negatives at the wire layer before the validator runs. A nonnegative-check test would exercise encoding/json's type machinery, not Helion code. |
| Port range boundary pair (1023 rejected / 1024 accepted / 65535 accepted / 65536 rejected) | The existing validation test covers 80 (privileged, rejected) and 70000 (out-of-range, rejected). The validator's checks (`Port < 1 \|\| Port > 65535` then `Port < 1024`) are two single-comparison gates; off-by-one flips would be caught by any of the existing tests. Adding boundary pairs would pin a laxity baseline without exercising new code. |
| Grace period upper bound boundary (30 min accepted, 30 min + 1 ms rejected) | Same reasoning ‚Äî existing test covers a clearly-oversize value; the boundary pair exercises the same single comparator. |
| GET /api/services/{job_id} when service enabled but prober hasn't Upserted yet | `ServiceRegistry.Get` returns `ok=false` for missing keys; handler translates to 404. The code path is identical to "job never existed" which is already tested in `handlers_services_test.go`. |
| Unhealthy ReportServiceEvent (Ready=false) via gRPC | `TestReportServiceEvent_UpsertsRegistry` tests Ready=true; the handler branches on `Ready` only for the event-name string (`service.ready` vs `service.unhealthy` in the slog line). Registry upsert and audit emission paths are identical. The prober integration test's flip sub-test exercises Ready=false end-to-end, so the RPC-level variant is covered by composition. |
| `IsService=true` disables runtime timeout enforcement | Runtime-level invariant ‚Äî feature-15/17 crossover. The runtime tests in `internal/runtime/` already cover the no-timeout path (`gpu_runtime_test.go` + peers); duplicating here would alarm for the same primitive. |
| Rust runtime parity for `IsService` | Deliberately deferred to backlog entry 20. Out of scope for this audit. |
| `probeService` exits silently on context cancel (no final event) | The spec documents this explicitly: "ReportServiceEvent only records live state, not 'the service is gone'." The terminal `serviceRegistry.Delete` rides the JobCompletionCallback. A test pinning silence would alarm if a regression added a final emit ‚Äî but that would be a feature change, not a regression against the spec. |
| Service registry `Delete` on job completion via `JobCompletionCallback` | Already covered implicitly by the callback wiring in `cmd/helion-coordinator/main.go` and the callback's own unit tests in workflow lifecycle. Adding a cross-package integration test here would re-alarm the callback's behaviour. |
| `advertiseAddr` carries the node's host:port for `/api/services/{id}` upstream URL construction | Trivial accessor ‚Äî the field is stamped once at startup and read unchanged. A test would pin getter/setter behaviour, not a regression target. |

Each evaluated against the audit framework bar ‚Äî **would a real
regression actually trip this test?** and **is the alarm
qualitatively different from what's already in place?** Only
T1 / T2 / T3 cleared both.

## Calibration note

Feature 17 is the first audited feature with a long-running
goroutine (`probeService`) that ships **without any test file at
all**. Prior features (11‚Äì16) had uneven coverage but every
production source file had at least one companion test. The
prober slipped through that filter because its wiring sits
between two gRPC surfaces ‚Äî easy to assume "the coordinator and
node tests cover their halves and the middle is just timing."

New rule of thumb: **every long-running goroutine needs at least
one integration test that drives multiple ticks.** The
edge-triggered emit is exactly the class of regression that
wouldn't show up in any static unit test ‚Äî you have to run the
loop long enough to observe transitions. For feature 17, 25 ms
tick √ó 5+ ticks √ó httptest real HTTP gives a robust signal in
under 200 ms.

Pattern across features 11‚Äì17 post-first-pass:
- 11: 4 passes (cross-backend contract saturation)
- 12: 4 passes (multiple production fixes)
- 13: 2 passes (splitter divergence)
- 14: 1 pass (event payload observer)
- 15: 1 pass (API boundary)
- 16: 3 passes (security + validator + audit-log observer)
- 17: 1 pass (entirely untested long-running goroutine)

Each feature's gap class tracks its shape. Feature 17's
distinctive shape is the goroutine; pass 1 closed that. The
recurring audit-log observer pattern also landed here as part
of T2.

Recommendation: feature 18 (dashboard ML module) next. It's the
consumer of every event pinned across audits 07, 09, 03, and
now 06 ‚Äî the dashboard audit will test the read side of those
contracts.
