## Medium

### ~~M2 ‚Äî Prometheus has no gauge for live service count~~ ‚úÖ Fixed
**File:** `internal/metrics/prometheus.go`

The coordinator exposes `helion_datasets_total` / `helion_models_total`
for the registry surface but had no equivalent for feature-17 service
endpoints. An operator running an inference-heavy cluster could not
dashboard live service count.

**Status:** Fixed. Added `ServiceCounter` interface
(`Count() int`), `helion_services_total` gauge, and
`SetServiceCounter` on the collector. `NewRegistry` takes a fifth
parameter (nil-safe). `cluster.ServiceRegistry` already satisfies
the interface via its existing `Count` method. Wired in
`cmd/helion-coordinator/main.go` alongside the registry counter.
Existing prometheus test signatures updated.

### ~~M3 ‚Äî `buildUpstreamURL` hand-rolled port parsing is IPv6-unsafe~~ ‚úÖ Fixed
**File:** `internal/api/handlers_services.go:88-108` (original code)

The upstream URL builder walked `NodeAddress` from the right and
stripped at the last colon. That works for IPv4 (`10.0.0.4:9090`)
but corrupts IPv6 literals (`[::1]:9090` ‚Üí it would strip at a
colon inside the address). Also the code had a slog.Warn inside
string parsing, which is noisy for a happy path.

**Status:** Fixed. Rewrote to use `net.SplitHostPort` and
`net.ParseIP`; IPv6 literals are re-wrapped in brackets when
composing the `http://[::1]:port/path` form. slog.Warn removed.

### M1 ‚Äî Workflow YAML cannot express service jobs ‚Üí **Deferred**
**File:** `internal/api/handlers_workflows.go:WorkflowJobSpec`

`SubmitRequest` grew a `Service` field; `WorkflowJobSpec` did not.
A user submitting `POST /workflows` with a `service` block inside a
job will have the field silently dropped (JSON decoder accepts
unknown fields). Inline `train ‚Üí register ‚Üí serve` DAGs are not
expressible today ‚Äî the user must run the service submit
out-of-band.

**Status:** Deferred. The plumbing change is small (three fields +
a matching assignment in the workflow ‚Üí `cpb.Job` copy path), but
the design question is real: service jobs never reach a terminal
state, so a DAG containing one doesn't cleanly cleanly finish. See
[`deferred/21-service-jobs-in-workflow.md`](../planned-features/deferred/21-service-jobs-in-workflow.md)
for the three design resolutions and revisit triggers.

**Status:** Fixed. Header comment now has a "Feature-flagged

### ~~L3 ‚Äî Probe response body not drained before Close~~ ‚úÖ Fixed
**File:** `internal/nodeserver/service_prober.go` ‚Äî `probe()` closure

Go's `http.Transport` requires the response body to be drained
before Close to reuse the underlying TCP connection. The original
code just called `Close` without draining, forcing a new TCP
handshake per probe tick. A healthy service with a 5s probe
interval sees ~12 handshakes/minute instead of zero.

**Status:** Fixed. `io.CopyN(io.Discard, resp.Body, 4<<10)` drains
up to 4 KiB before Close ‚Äî enough to recycle the connection,
bounded so a misbehaving `/healthz` that streams megabytes cannot
slow the prober down.

### L1 ‚Äî No rate limit on `ReportServiceEvent` RPC ‚Üí **Deferred**
**File:** `internal/grpcserver/handlers.go:ReportServiceEvent`

The node-RPC rate limiter covers `Register` and `ReportResult` but
not the new feature-17 RPC. A compromised or malicious node could
flap readiness and flood the audit log / Prometheus scrape.

**Status:** Deferred. The emission path is edge-triggered
(one event per state flip, not per probe tick), so a legitimate
service can't drive the volume. A malicious node would hit the
cross-node-poison check on every spurious event, which already
logs a security violation. The right time to add an explicit
limiter is when the node-RPC limiter grows a per-RPC-type matrix
(one line at that point), or when production traffic shows the
gap bites. See
[`deferred/22-report-service-event-rate-limit.md`](../planned-features/deferred/22-report-service-event-rate-limit.md).

## Test Gaps

### ~~T2 ‚Äî No test for cross-node poison rejection on ReportServiceEvent~~ ‚úÖ Fixed
**File:** `internal/grpcserver/report_service_event_test.go` (new)

The handler's cross-node-poison guard was landed in the original
commit but was only exercised by eyeball on the existing
`ReportResult` tests. Added:

- `TestReportServiceEvent_UpsertsRegistry` ‚Äî happy path, the correct
  node pushes a ready event and the registry reflects it.
- `TestReportServiceEvent_CrossNodeMismatch_Rejected` ‚Äî a different
  node tries to claim the first node's service job; `PermissionDenied`
  is returned, the registry stays empty, and `LogSecurityViolation`
  fires on the audit mock.
- `TestReportServiceEvent_NilRegistry_SoftAccepts` ‚Äî the soft-accept
  path when `SetServiceRegistry` was not called; RPC returns ok so
  the node prober isn't stuck retrying against a deployment that
  doesn't enable services.

### T1 ‚Äî No end-to-end integration test for service dispatch ‚Üí **Deferred**
**File:** (no corresponding file)

`tests/integration/` covers node registration, heartbeat, job
dispatch, and result reporting over real mTLS. A service

**Status:** Deferred. The unit tests (registry, handler,

and would duplicate the integration-test coverage. See
[`deferred/23-service-integration-e2e.md`](../planned-features/deferred/23-service-integration-e2e.md).

## Items considered and dismissed (not findings)

| Item | Why it's fine |
|------|--------------|
| Service bypasses timeout enforcement | By design ‚Äî a service is supposed to run until cancelled. `IsService` flag on `RunRequest` comes from the coordinator-signed `Job.Service`, not from the node, so a compromised node can't flip it. |
| Probe URL is HTTP not HTTPS | Per spec ‚Äî the probe is loopback-only (`127.0.0.1`). mTLS-to-self is complexity without payoff. |
| Services tracked in memory only | Intentional ‚Äî live state, not history. A coordinator restart is re-populated by the next probe tick (edge-triggered emission recovers within one probe interval because the prober's `haveLastReady` flag starts false per-job). |
| `Cancel` on a service job does not explicitly `serviceRegistry.Delete` | The `JobCompletionCallback` wired in main.go fires on every terminal transition (including cancelled), which calls `serviceRegistry.Delete`. Single cleanup path covers both job-completed and job-cancelled. Verified by reading the callback wiring; unit coverage on the delete path lives in `service_registry_test.go`. |
| `jobFromDispatch` on the node doesn't copy `Service` | The function only exists to hand a `cpb.Job` to the stager for artifact staging; the stager doesn't care about Service. The prober reads `req.Service` directly from the `DispatchRequest`. |
