## Medium

### ~~M1 — Registry operations invisible to Prometheus~~ ✅ Fixed
**File:** `internal/metrics/prometheus.go`

The coordinator exposes `helion_jobs_total`, `helion_running_jobs`,
`helion_healthy_nodes`, etc. but had no equivalent for dataset or
model counts. Operators running a registry-enabled coordinator would
see zero observability into registry growth — the closest signal was
the `dataset.registered` / `model.registered` audit events, which
are per-event and not suitable for dashboarding cardinality or
capacity planning.

**Status:** Fixed. Added `helion_datasets_total` and
`helion_models_total` pull-based gauges. The collector accepts a
new `RegistryCounter` interface
(`CountDatasets(ctx) / CountModels(ctx)`); the BadgerStore
implementation walks the key prefix with `PrefetchValues=false` so
the count pays an LSM-level scan cost only, not a JSON-decode cost.
`NewRegistry` now takes a fourth argument — pass `nil` on
deployments that don't enable the registry and the gauges are
omitted from scrape output entirely (no misleading zeros).
Two new tests cover both the emitted and the omitted path.

### ~~M2 — SECURITY.md did not cover the new registry surface~~ ✅ Fixed (pre-audit)
**File:** `docs/SECURITY.md`

Standing rule (per user memory
`feedback_always_tests_and_hybrid_security.md`): every slice gets a
SECURITY.md integration. Step 6 landed without one — the initial
step-6 commit was through the handlers but left SECURITY.md silent
on the new endpoints, validators, and rate-limiter.

**Status:** Fixed in commit `1ef14d2` (immediately before this
audit pass began). SECURITY.md section 5 now has a Registry

k8s-shaped tags), and the deliberate deferrals (no URI existence
check, no cascade delete, flat authorization). Recorded here so the
audit reflects the full step-6 delta, not just the code-side fixes.

### M3 — No cascade or warning on dataset delete with dependent models → **Deferred**
**File:** `internal/registry/badger.go:121` — `DeleteDataset`

`DeleteDataset` removes a dataset entry unconditionally. Models
that reference the dataset via `source_dataset` keep their lineage
pointer but the pointer now dangles: `GET /api/models/m/v1`
returns a model whose `source_dataset` refers to a name+version
that no longer resolves via `GET /api/datasets/...`.

**Judgement:** not a bug. The spec explicitly treats lineage as
*soft* — a historical audit trail, not a foreign-key constraint.
Tightening this would require one of:

1. Reject delete when any model references the dataset (requires
   scanning every model per delete; also blocks legitimate GDPR
   / retention deletes).
2. Cascade-delete dependent models (dangerous — silently removes
   artifacts that may be in production serving paths).
3. Mark lineage as "dangling" on read (requires materialising the
   join at GET time, changing the response shape).

None of these is obviously correct; the lowest-regret position is
"lineage is a hint, clients verify independently." Deferred to
[`docs/planned-features/deferred/README.md`](../planned-features/deferred/README.md)
under *ML Pipeline → Registry lineage enforcement*.

### M4 — `source_job_id` is not validated against the JobStore → **Deferred**
**File:** `internal/api/handlers_registry.go:265` — `handleRegisterModel`

A caller can register a model with `source_job_id: "train-1"`
where no such job has ever existed. The registry accepts the
pointer without round-tripping it against the JobStore.

**Judgement:** same soft-pointer rationale as M3. Enforcing would
require the registry package to import the JobStore, which
collapses the current clean separation (registry knows nothing
about job lifecycle). The check is also race-prone: the job can
be GC'd after validation but before the model is consumed. The
operational cost of accepting stale `source_job_id` is low — a
broken UI link at worst — so the soft pointer is kept. Deferred
to the same backlog entry as M3.

## Low

### L1 — `ListDatasets` / `ListModels` load every entry before paginating → **Deferred**
**File:** `internal/registry/badger.go:105,221`

`ListDatasets` calls `loadAllDatasets` (full prefix scan + JSON
decode for every entry) and *then* sorts by `CreatedAt` descending
and slices to the requested page. Cost grows O(n) with the total
registered count regardless of requested page size.

**Judgement:** not a real DoS. The handler is behind the
registry rate limiter (2/s burst 30 per subject) and the expected
registry size for a minimal-ML deployment is ≤ 1000 entries.
Even at 100k entries the scan is sub-50 ms with BadgerDB's LSM
layout. The fix (secondary index on `CreatedAt` or cursor-based
pagination) is a meaningful scope increase that earns nothing
at current traffic. Deferred until a real operator reports the
need. Filed in the backlog as *Registry indexed listing*.

### L2 — `PrefetchValues=false` in Count, unused elsewhere → **Acknowledged**
**File:** `internal/registry/badger.go`

`countPrefix` disables value prefetch; `loadAllDatasets` /
`loadAllModels` don't. That's correct (they need the values to
decode), just an asymmetry worth noting. No action — this is a
style observation, not a finding.

### L3 — Integration-test coverage stops at the API mux → **Deferred**
**File:** *(no corresponding file)*

`internal/api/handlers_registry_test.go` drives the HTTP
handlers via `do(srv, ...)` — an in-process request through the
ServeMux with a shared `newServer` helper. There is no test under
`tests/integration/` that runs a real coordinator binary with
mTLS + a real BadgerDB + a registered dataset. The API-layer tests
cover validators, rate limiter, event emission, audit hooks, and
error paths; the gap is purely "does the wiring in main.go
actually expose the endpoints."

**Judgement:** low value, moderate cost. The existing
`tests/integration` harness is shaped around gRPC node
registration and workflow dispatch — a registry integration test
would be new boilerplate for an HTTP-only surface that the
handler tests already exercise end-to-end within a single
process. Deferred to the backlog; if a wiring regression ever
lands it will be caught by the first registry-using feature
(step 10's iris example).

## Items considered and dismissed (not findings)

These were evaluated during the audit pass but neither fixed nor
deferred — they are intentional design decisions that were already
documented:

| Item | Why it's fine |
|------|--------------|
| `URI` is not existence-checked at register time | Spec'd; would require artifact-store creds on the coordinator. Documented in SECURITY.md §5. |
| Flat authz (any authenticated user can delete any entry) | Matches the rest of the API. Tightening is a product policy call, not an audit finding. |
| `LatestModel` uses wall-clock `CreatedAt` rather than semver | Spec'd; `Get(name, exact-version)` is the escape hatch. |
| Shared BadgerDB with jobs / workflows / audit | Deliberate — separate DB would be operational overhead for no isolation benefit. Recorded in `internal/registry/badger.go` package doc. |
| No per-job UI linkback from `source_job_id` | Dashboard concern, not a registry concern. Step 8 (dashboard ML module) owns this. |
