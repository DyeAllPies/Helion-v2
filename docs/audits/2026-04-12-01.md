## High

### ~~H1 — Dispatcher TLS skips server certificate verification~~ ✅ Fixed
**File:** `cmd/helion-coordinator/main.go:281-283` — dispatch TLS config

```go
// Accept any server cert from nodes (same as RequireAnyClientCert logic)
dispatchTLS.InsecureSkipVerify = true
nodeDispatcher := cluster.NewGRPCNodeDispatcher(dispatchTLS)
```

When the coordinator opens a gRPC connection to a node to dispatch a job,
it sets `InsecureSkipVerify = true`. This means the coordinator trusts
**any** server certificate (or none at all) presented by the node.
A man-in-the-middle on the coordinator-to-node network segment can:

- Intercept all dispatched job definitions (commands, args, env vars which
  may contain secrets).
- Forge `DispatchAck` responses, causing the coordinator to believe jobs
  were accepted when they were not (or vice versa).
- Redirect jobs to an attacker-controlled process.

The comment references "RequireAnyClientCert logic" — the intent is that
nodes are already authenticated via mTLS in the reverse direction
(node→coordinator). However, this does not protect the forward direction
(coordinator→node). The coordinator should verify the node's server
certificate against the CA it issued during registration, or at minimum
verify the fingerprint it pinned at registration time.

**Status:** Fixed. Removed `InsecureSkipVerify = true`. The dispatch TLS config
now verifies node certificates against the coordinator's internal CA (RootCAs
pool set by `RawTLSConfig`). Added `TestDispatchTLS_VerifiesNodeCert` (T1).

### ~~H2 — WebSocket authentication via URL query parameter~~ ✅ Fixed
**File:** `dashboard/src/app/core/services/websocket.service.ts:43` —
`_connect()`

```typescript
const url = token ? `${baseUrl}?token=${encodeURIComponent(token)}` : baseUrl;
```

The JWT bearer token is attached to the WebSocket URL as a query parameter.
This is the only available mechanism for browser-initiated WebSocket auth
(the `WebSocket` constructor does not support custom headers), but it
creates several exposure vectors:

- **Server access logs**: The full URL including `?token=...` is logged by
  reverse proxies (nginx, Envoy, ALB) and web servers by default.
- **Browser history**: The URL is stored in the browser's history and can
  be extracted by other users of the same machine.
- **Referer leakage**: If the page navigates to an external link while the
  WebSocket URL is in the address bar, the token leaks via the `Referer`
  header (mitigated in this app by CSP and SPA routing, but fragile).

The server-side handler (`internal/api/handlers_ws.go`) correctly extracts
the token from `r.URL.Query().Get("token")`, so the mechanism works — the
issue is the exposure surface.

**Status:** Fixed. WebSocket connects without token in URL. Client sends
`{"type":"auth","token":"..."}` as the first frame after `onopen`. Server
validates the token and replies `{"type":"auth_ok"}` before streaming data.
Token never appears in URLs, server logs, or browser history.

## Medium

### ~~M1 — Raw `err.Error()` leaked in admin token-issue endpoint~~ ✅ Fixed
**File:** `internal/api/handlers_admin.go:94` — `handleIssueToken`

```go
writeError(w, http.StatusBadRequest, "invalid JSON: "+err.Error())
```

The `json.Decoder` error message is concatenated directly into the HTTP
response. Go's JSON decoder includes type names, struct field paths, and
expected-vs-actual type information in its errors. An attacker probing the
`POST /admin/tokens` endpoint with malformed JSON payloads can infer
internal struct layouts and Go version details.

Other handler entry points (`handlers_jobs.go:117`) correctly use a generic
`"invalid request body"` message. This is the only remaining call site
that leaks decoder internals.

**Status:** Fixed. Replaced with generic `"invalid request body"` message.
Raw error logged at debug level via `slog.Debug`.

### ~~M2 — Dashboard components expose raw error messages to users~~ ✅ Fixed
**Files:**
- `dashboard/src/app/features/audit/audit-log.component.ts:193`
- `dashboard/src/app/features/jobs/job-detail.component.ts:352`
- `dashboard/src/app/features/jobs/job-list.component.ts:272`
- `dashboard/src/app/features/nodes/node-list.component.ts:285`
- `dashboard/src/app/features/metrics/cluster-metrics.component.ts:288`

All five components interpolate `(err as Error).message` directly into the
UI error banner. Example:

```typescript
this.error = `Failed to load audit log: ${(err as Error).message}`;
```

HTTP error responses from the API (including the M1 raw JSON error above)
and network-level errors (DNS failures, TLS handshake details) are rendered
verbatim. This is information disclosure to any authenticated dashboard
user and could reveal internal hostnames, API paths, or infrastructure
details.

**Status:** Fixed. All 5 components now display generic user-facing messages
and log the raw error to `console.error` for developer diagnostics.

### ~~M3 — `VerifyCertificateWithKEM` is a no-op stub~~ ✅ Fixed
**File:** `internal/pqcrypto/hybrid.go:231-238` — `VerifyCertificateWithKEM`

```go
func VerifyCertificateWithKEM(_ *x509.Certificate, _ *KEMPublicKey) error {
    // TODO(Phase 5): Implement ML-DSA (Dilithium-3) signature verification.
    return nil
}
```

This function unconditionally returns `nil`, meaning any certificate
passes ML-DSA verification. The function is currently not called in
production code (only in tests), so it is **not exploitable today**.
However, `VerifyMLDSASignature` in `mldsa.go:175` *is* fully implemented
and tested — the gap is that `VerifyCertificateWithKEM` does not delegate
to it. If a future caller wires up `VerifyCertificateWithKEM` in the TLS
handshake path (as the function's name and doc suggest), it would silently
skip ML-DSA verification.

**Status:** Fixed. `VerifyCertificateWithKEM` now accepts `*MLDSAPublicKey`
and delegates to `VerifyMLDSASignature`. When `caPub` is nil (ML-DSA not
enabled), verification is skipped for backwards compatibility. Tests cover
valid cert, missing extension, and nil-pubkey paths.

### ~~M4 — `extractJTIUnchecked` uses `ParseUnverified` on stored token~~ ✅ Fixed
**File:** `internal/auth/jwt.go:200,243-244` — `RotateRootToken` /
`extractJTIUnchecked`

```go
if oldJTI, err := extractJTIUnchecked(string(existing)); err == nil {
    _ = tm.RevokeToken(ctx, oldJTI)
}
```

During root-token rotation, the old token is read from BadgerDB and parsed
with `jwt.Parser.ParseUnverified` to extract the JTI for revocation. The
function's own doc comment (lines 237-242) acknowledges this is safe only
if the token comes from trusted storage. The security comment is thorough
and the blast radius is limited (the JTI is only used for revocation, not
authentication), so this is not directly exploitable.

However, if BadgerDB is corrupted (disk error, incomplete write) the
function could extract a garbage JTI and attempt to revoke it — a no-op
but a confusing audit trail. More importantly, the pattern sets a precedent
for `ParseUnverified` usage that could be copied by future contributors.

**Status:** Fixed. `RotateRootToken` now calls `tm.ValidateToken` instead
of `extractJTIUnchecked`. If validation fails (secret regenerated), revocation
is skipped — the old token is already invalid. Eliminates `ParseUnverified`
from the rotation path.

## Low

### ~~L1 — HTTP server timeouts may not prevent slow-request DoS~~ ✅ Fixed
**File:** `internal/api/server.go:189-194`

```go
hsrv := &http.Server{
    Handler:      s.mux,
    ReadTimeout:  10 * time.Second,
    WriteTimeout: 10 * time.Second,
    IdleTimeout:  60 * time.Second,
}
```

`ReadTimeout` and `WriteTimeout` apply to the entire connection, not
individual requests on a keep-alive connection. A Slowloris-style attacker
can open many connections, send headers slowly (one byte every 9 seconds),
and hold slots. The 10-second `ReadTimeout` provides partial protection —
the connection will be closed after 10 seconds regardless — but the
`IdleTimeout` of 60 seconds allows a keep-alive connection to sit idle
between requests, and the next request on that connection resets the read
deadline.

In practice, `ReadTimeout = 10s` is effective against most slow-read
attacks. The risk is low because the coordinator is expected to sit behind
a reverse proxy (nginx, Envoy) that enforces its own timeouts.

**Status:** Fixed. Added `ReadHeaderTimeout: 5 * time.Second` to the HTTP
server config, specifically designed to counter Slowloris-style attacks.

### ~~L2 — WebSocket `CheckOrigin` allows missing `Origin` header~~ ✅ Fixed
**File:** `internal/api/server.go:122-136` — WebSocket upgrader

Requests without an `Origin` header are allowed through. This is
intentional for CLI/API tool access (curl, wscat). The code comment at
lines 122-125 now documents the design rationale:

```go
// Reject cross-origin WebSocket connections. Browsers always send an
// Origin header on WebSocket upgrades; we compare its host component
// against the request Host so that only same-origin pages can connect.
// curl / native clients that omit Origin are allowed through.
```

**Status:** Fixed — comment documents the intentional design decision.
No code change was needed.

### Execution batches (2026-04-12)

| # | Batch | Findings | Risk | Primary files |

**Why this order:** Batch 1 closes the only network-exploitable gap
(MITM on dispatch). Batch 2 reduces token exposure surface. Batch 3 is
a quick string-replacement across a few files. Batches 4-6 are hardening
with no urgent exploit path.

### H1 — Dispatch TLS verification

1. In `cmd/helion-coordinator/main.go`, remove `dispatchTLS.InsecureSkipVerify = true`.
2. Set `dispatchTLS.RootCAs` to the coordinator's CA certificate pool.
3. Add a `VerifyPeerCertificate` callback that checks the presented
   certificate's SHA-256 fingerprint against the pinned fingerprint in
   the `Registry` for the target node address.
4. Add `TestDispatchTLS_VerifiesNodeCert` integration test.

### H2 — WebSocket first-message auth

1. In `websocket.service.ts`, connect without `?token=` in the URL.
2. After `onopen`, send `{"type":"auth","token":"<jwt>"}` as the first
   frame.
3. In `handlers_ws.go`, buffer the first frame, extract and validate the
   token before subscribing to the data stream. Close with 4001 if
   auth fails.
4. Update E2E tests that connect via WebSocket.

### M1 — Admin error sanitization

Replace line 94 of `handlers_admin.go`:
```go
writeError(w, http.StatusBadRequest, "invalid request body")
```
Log the raw error with `s.log.Debug(...)`.

### M2 — Dashboard error sanitization

In all five components, replace the interpolated error with a generic
message. Log `err` to `console.error` for developer diagnostics.

### M3 — Implement ML-DSA verification in `VerifyCertificateWithKEM`

Implement `VerifyCertificateWithKEM` in `hybrid.go` by delegating to the
existing `VerifyMLDSASignature` function:

1. Accept `*MLDSAPublicKey` (or extract it from `*KEMPublicKey` if
   bundled) as the CA's ML-DSA public key.
2. Call `VerifyMLDSASignature(cert, caPub)` and return the result.
3. Update `TestVerifyCertificateWithKEM_NoError` to verify that a cert
   **with** a valid ML-DSA extension passes, and a cert **without** one
   fails.
4. Remove the `TODO(Phase 5)` comment.

### M4 — Token rotation

In `RotateRootToken`, attempt `tm.ValidateToken(ctx, string(existing))`
before extracting the JTI. If validation fails, skip revocation (the
token is already invalid under the new secret). Remove the
`extractJTIUnchecked` call from the rotation path.
