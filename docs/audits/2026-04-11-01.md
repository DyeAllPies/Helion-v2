## Medium

### ✅ M1 — Fire-and-forget audit writes use `context.Background()` *(fixed 2026-04-11)*
**File:** `internal/cluster/job_submit.go:45-49` — `JobStore.Submit` (and the
parallel pattern in `internal/cluster/job_transition.go`)

```go
go func() {
    _ = s.persister.AppendAudit(context.Background(),
        "job.submitted", "coordinator", j.ID,
        fmt.Sprintf("command=%q", j.Command))
}()
```

The audit-write goroutine is detached from the request context and has no
timeout. If the persister stalls (BadgerDB compaction storm, disk pressure,
hung underlying filesystem), these goroutines accumulate and the error is
swallowed. They also outlive the parent request and the coordinator's
graceful-shutdown window — `srv.Shutdown` cannot wait for them, so audit
events for the very last requests can be lost on SIGTERM.

**Fix:** Introduced `appendAuditAsync` / `persistNodeAsync` helpers on both
`JobStore` and `Registry` that wrap each fire-and-forget write in a 5s
`context.WithTimeout` and log at warn on failure. An `auditWG sync.WaitGroup`
tracks in-flight goroutines; `JobStore.Close(timeout)` and `Registry.Close(timeout)`
drain them during shutdown, and the coordinator main now calls both before
returning. All six audit/persist call sites across `job_submit.go`,
`job_transition.go`, `registry_register.go`, `registry_revoke.go`,
`registry_heartbeat.go` were migrated. Tests:
`TestJobStore_Close_DrainsHangingAudit` and
`TestJobStore_AuditTimeout_BoundsHangingWrite` in
`internal/cluster/audit_drain_test.go`.

### ✅ M2 — Rate-limiter map unbounded for up to 5 minutes *(fixed 2026-04-11)*
**File:** `internal/ratelimit/limiter.go:96-114` (`getOrCreate`) +
`cmd/helion-coordinator/main.go:252-267` (GC goroutine)

`getOrCreate` inserts a new `limiterEntry` for every distinct `nodeID` it
sees. The only eviction path is the GC goroutine in `main.go`, which runs on
a `5 * time.Minute` ticker. An attacker who can spoof or churn `nodeID`
values on inbound gRPC requests can insert millions of entries between GC
cycles before the next sweep evicts them. Each entry is small but the map
itself is unbounded — there is no insertion-time cap.

**Why this is reachable:** the limiter is consulted from the gRPC interceptor
**before** authentication completes (rate limiting must run pre-auth to
protect the auth path itself), so spoofed node IDs from unauthenticated
peers count.

**Fix:** Added `MaxLimiters = 10_000` cap enforced inside `getOrCreate`
under the write lock. On overflow, `evictOldestLocked` removes the entry
with the smallest `lastSeenNano` — stale, genuine nodes take priority
over new arrivals from an attacker. `capEvictions` atomic counter exposed
via `CapEvictions()` for metrics. Test: `TestNodeLimiter_CapOnOverflow`
in `internal/ratelimit/limiter_test.go`.

### ✅ M3 — Helm pod securityContext is missing hardening fields *(fixed 2026-04-11)*
**File:** `deploy/helm/values.yaml:44-47, 81-83`

Both `coordinator.securityContext` and `node.securityContext` set only
`runAsNonRoot`, `runAsUser`, and (coordinator only) `fsGroup`. They are
missing every other recommended pod-security field:

- `readOnlyRootFilesystem: true`
- `allowPrivilegeEscalation: false`
- `capabilities: { drop: ["ALL"] }`
- `seccompProfile: { type: RuntimeDefault }`

The coordinator only writes to its PVC mount and the node only needs `/tmp`
(if at all), so a read-only root is feasible. Without these fields the
container would survive container-escape primitives that pod-security
admission controllers expect to block.

**Fix:** Added `seccompProfile: RuntimeDefault` to the pod-level
`securityContext` and introduced a new container-level
`containerSecurityContext` key with `readOnlyRootFilesystem: true`,
`allowPrivilegeEscalation: false`, and `capabilities: drop: [ALL]`.
Both coordinator and node pod templates now render container
securityContext and mount an `emptyDir` at `/tmp` so the Go stdlib's
scratch writes still succeed under read-only root. Validated with
`helm template` — 10 manifests render without error.

### ✅ M4 — Dockerfiles lack `HEALTHCHECK` directive *(fixed 2026-04-11)*
**File:** `Dockerfile.coordinator`, `Dockerfile.node`

Neither image declares `HEALTHCHECK`. Kubernetes uses its own probes so the
gap is invisible there, but `docker-compose.yml` and any non-k8s deployment
loses container-level liveness signal — an unresponsive coordinator stays
`Up` in `docker ps` and downstream orchestrators (Swarm, Nomad, plain
Docker) cannot react.

**Fix:** Added `HEALTHCHECK --interval=10s --timeout=3s --start-period=5s
--retries=3` to both Dockerfiles. Coordinator probes
`wget -qO- http://localhost:8080/healthz`; node probes `nc -z localhost 8080`
since it only speaks gRPC. Kubernetes probes override these in the Helm
chart so there is no double-probe overhead.

### ✅ M5 — Mutable tag `busybox:1.36` in init container *(fixed 2026-04-11)*
**File:** `deploy/helm/templates/node-daemonset.yaml:27`

```yaml
- name: wait-for-coordinator
  image: busybox:1.36
```

`busybox:1.36` is a mutable tag — the underlying digest changes when
upstream rebuilds the minor line. A future rebuild can pull in CVE
regressions or behavioural changes silently. This is the only image in the
chart not parameterised through `values.yaml`.

**Fix:** Parameterised through `.Values.node.initImage.{repository,tag,pullPolicy}`
(default `busybox:1.36`) so operators can pin a digest in their own values
without forking the chart. Also runs under the node's
`containerSecurityContext` now so the wait-loop inherits the same
hardened profile as the main container.

## Low

### ✅ L1 — `/metrics` is unauthenticated when a Prom handler is wired *(fixed 2026-04-11)*
**File:** `internal/api/server.go:157-163`

```go
if s.promHandler != nil {
    s.mux.Handle("GET /metrics", s.promHandler)
} else {
    s.mux.HandleFunc("GET /metrics", s.authMiddleware(s.handleGetMetrics))
}
```

The fall-through with no auth is intentional (commented as such) so
Prometheus can scrape without a token, but the metrics expose cluster
topology, node counts, job throughput, and queue depth — useful
reconnaissance for an attacker on the same network. The current design
relies entirely on a NetworkPolicy or firewall to restrict access; nothing
in the application enforces it.

**Fix:** Added `deploy/helm/templates/coordinator-networkpolicy.yaml`
gated by `.Values.coordinator.networkPolicy.enabled` (default false so
existing deploys are unchanged). When enabled, only pods matching
`metricsFrom` (default: `kubernetes.io/metadata.name=monitoring`) can
reach the HTTP port for scraping, and only `apiFrom` selectors (default
empty — no in-cluster API ingress) can reach HTTP/gRPC. Operators who
rely on reverse-proxy access keep the policy disabled; operators who
want the defense-in-depth enable it in their values overlay.

### ✅ L2 — JSON response encoder errors silently dropped *(fixed 2026-04-11)*
**File:** `internal/api/handlers_admin.go:73, 138, 183` and several handlers
in `handlers_jobs.go` / `handlers_cluster.go`

```go
_ = json.NewEncoder(w).Encode(resp)
```

By the time `Encode` runs, the status header is already flushed, so there
is little to recover — but the failure is invisible to operators. A
truncated response from an admin token-issue endpoint would be especially
nasty to debug.

**Fix:** Added `writeJSON(w, handler, v)` helper in `internal/api/helpers.go`
that logs at warn with the handler name on encode failure. All 10
call sites across `handlers_admin.go`, `handlers_cluster.go`, and
`handlers_jobs.go` migrated. `writeError` also updated to log instead of
silently swallow.

### ✅ L3 — Container base images not digest-pinned *(accepted as dev tradeoff; documented 2026-04-11)*
**File:** `Dockerfile.coordinator:1,10`, `Dockerfile.node:1,10`

`golang:1.26-alpine` and `alpine:3.20` are mutable tags. Same class of
issue as M5 but for the application images themselves. Lower severity
because the build pipeline rebuilds frequently and the images are
regenerated from source on each release.

**Resolution:** Left mutable tags in the committed Dockerfiles so local
dev builds stay ergonomic. Release builds should pin to digests in CI;
this is captured as a Deployment Note below rather than a code change.

### ✅ L4 — `HELION_ROTATE_TOKEN` truthiness check is loose *(fixed 2026-04-11)*
**File:** `cmd/helion-coordinator/main.go:138`

```go
if envOr("HELION_ROTATE_TOKEN", "true") != "false" { ... rotate ... }
```

Anything other than the literal string `false` rotates. Operators who set
`HELION_ROTATE_TOKEN=0` or `HELION_ROTATE_TOKEN=no` expecting it to disable
rotation will get the opposite. Default-secure (errs on the side of
rotating) but surprising.

**Fix:** `HELION_ROTATE_TOKEN` is now parsed with `strconv.ParseBool` in
`cmd/helion-coordinator/main.go`. Unset = default-rotate; unknown values
exit the process at startup with an explicit error message. Standard
boolean spellings (`0`/`1`, `true`/`false`, `yes`/`no`, `T`/`F`) now all
behave as expected.

## Deployment & Configuration Notes

- **`/metrics` auth** is enforced at the network layer. The optional
  `coordinator.networkPolicy` Helm block (disabled by default) restricts
  ingress to a configurable namespace selector. Operators not using
  NetworkPolicies must put the coordinator behind a reverse proxy that
  enforces authn/z on `/metrics` itself.
- **Container base images use mutable tags** (`golang:1.26-alpine`,
  `alpine:3.20`, `busybox:1.36`) for local-dev convenience. Release CI
  should resolve these to immutable digests before pushing to a
  production registry.
- **No config-file support** — all configuration is via env vars (see
  `cmd/helion-coordinator/main.go` `envOr` calls).
- **BadgerDB is single-writer** — coordinator Deployment uses
  `strategy: Recreate` so two pods never hold the PVC simultaneously.
- **Graceful shutdown timeout = 15s** for HTTP/gRPC, plus a 5s
  audit-drain budget for `JobStore.Close` and `Registry.Close` (see the
  M1 fix). In the worst case, shutdown can take 20s.
- **readOnlyRootFilesystem + `/tmp` emptyDir**: both coordinator and
  node pods run with a read-only root; the Go stdlib's temp-file path
  is backed by an `emptyDir` so scratch writes still work.
